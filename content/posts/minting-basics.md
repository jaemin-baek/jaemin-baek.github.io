![민팅 이해하기](/images/minting-basics-cover.png)

블록체인을 공부하다 보면 `mint`, `minting`, `민팅 완료`, `NFT 발행` 같은 표현을 자주 본다.

처음에는 민팅을 "NFT를 사는 것" 정도로 이해하기 쉽다. 실제로 많은 서비스에서 사용자는 민팅 페이지에 들어가고, 지갑을 연결하고, 결제하듯 버튼을 누른다. 결과로 NFT가 내 지갑에 들어온다.

하지만 민팅은 단순 구매 버튼이라기보다, 블록체인 위에 새 토큰 기록을 만드는 트랜잭션에 가깝다.

이 글은 민팅을 깊은 컨트랙트 구현 관점으로 설명하는 글은 아니다. [[web3-service-basics|Web3 기반 서비스]]에서 봤던 지갑, 서명, 트랜잭션, 스마트 컨트랙트 흐름 위에 "새로 발행한다"는 감각을 얹어보는 초심자용 메모에 가깝다.

## 먼저 한 문장으로 보기

민팅을 한 문장으로 줄이면 이렇게 볼 수 있다.

```text
민팅은 블록체인 위에서 새 토큰이나 새 소유권 기록을 발행하는 동작이다.
```

여기서 중요한 단어는 세 개다.

- 새 기록
- 발행
- 트랜잭션

민팅은 이미 있는 토큰을 다른 사람에게 보내는 것과 다르다. 없는 것을 새로 만들거나, 아직 아무에게도 배정되지 않은 토큰을 특정 주소에 배정하는 쪽에 가깝다.

그래서 민팅을 처음 볼 때는 아래 질문을 잡는 편이 좋다.

```text
이 트랜잭션은 무엇을 새로 만들고,
그 결과를 어느 주소에 기록하는가?
```

## 민팅은 화면 이벤트가 아니라 컨트랙트 실행이다

사용자 입장에서는 민팅이 버튼 하나처럼 보인다.

```text
Mint 버튼
-> 지갑 팝업
-> 확인
-> 민팅 완료
```

하지만 내부에서는 보통 스마트 컨트랙트 함수가 실행된다.

![민팅 트랜잭션 흐름](/images/minting-flow.png)

조금 더 풀면 이런 흐름이다.

```text
사용자가 민팅 페이지를 연다.
지갑을 연결한다.
서비스가 민팅할 컨트랙트와 수량, 가격을 준비한다.
사용자가 Mint 버튼을 누른다.
지갑이 트랜잭션 내용을 보여준다.
사용자가 확인하면 지갑이 서명한다.
서명된 트랜잭션이 네트워크로 전송된다.
스마트 컨트랙트의 mint 함수가 실행된다.
컨트랙트 상태에 새 토큰 기록이 반영된다.
서비스나 지갑 화면이 결과를 다시 보여준다.
```

여기서 지갑은 결제사처럼 민팅을 대신 처리하는 주체가 아니다. [[non-custodial-wallet|논커스토디얼 지갑]]이라면 지갑은 사용자의 개인키로 트랜잭션에 서명하는 도구에 가깝다.

실제 규칙은 컨트랙트가 가진다. 누가 민팅할 수 있는지, 몇 개까지 가능한지, 가격이 얼마인지, 이미 품절됐는지 같은 조건은 컨트랙트 코드나 그 컨트랙트가 참조하는 상태에 의해 결정된다.

## NFT 민팅은 새 NFT 소유권을 기록하는 일이다

민팅이라는 단어는 NFT 문맥에서 가장 자주 들린다.

NFT 민팅은 보통 새 NFT를 만들고, 그 NFT의 소유자를 어떤 주소로 기록하는 흐름을 뜻한다. 예를 들어 어떤 ERC-721 컨트랙트가 `tokenId = 42`인 NFT를 새로 만들고, 그 소유자를 내 주소로 기록할 수 있다.

아주 단순화하면 이런 상태 변화다.

```text
mint(tokenId = 42, owner = 내 주소)

before:
tokenId 42는 아직 없음

after:
tokenId 42의 owner는 내 주소
```

블록 탐색기나 마켓플레이스에서는 이 결과를 보고 "이 주소가 이 NFT를 가지고 있다"고 보여준다.

여기서 중요한 점은 이미지 파일 자체가 지갑 안으로 들어오는 것이 아니라는 점이다. NFT는 보통 토큰 ID와 소유권 기록이 체인 위에 있고, 이미지나 메타데이터는 별도 URI를 통해 참조되는 구조가 많다.

그래서 NFT 민팅을 볼 때는 두 가지를 나눠야 한다.

| 구분 | 의미 |
| --- | --- |
| 온체인 소유권 | 어떤 주소가 어떤 tokenId를 소유하는지 |
| 메타데이터 | 이름, 설명, 이미지 URI 같은 표시 정보 |

민팅이 성공했다는 것은 보통 온체인 소유권 기록이 생겼다는 뜻이다. 이미지가 언제 보이는지, reveal이 언제 되는지, 메타데이터가 바뀔 수 있는지는 프로젝트 설계에 따라 다르다.

## 토큰 민팅도 있다

민팅은 NFT에만 쓰는 말이 아니다.

ERC-20 같은 대체 가능한 토큰에서도 mint라는 표현을 쓴다. 이 경우에는 특정 주소의 토큰 잔액을 새로 늘리는 동작에 가깝다.

예를 들어 어떤 토큰 컨트랙트가 내 주소에 100개를 민팅한다고 해보자.

```text
before:
내 주소의 ABC 잔액 = 0
총 공급량 = 1,000

after:
내 주소의 ABC 잔액 = 100
총 공급량 = 1,100
```

NFT 민팅은 보통 "고유한 tokenId 하나를 누구에게 배정했는가"를 본다. ERC-20 민팅은 "어떤 주소의 잔액과 전체 공급량이 얼마나 늘었는가"를 본다.

둘 다 새 토큰 기록을 만든다는 점에서는 비슷하지만, 토큰의 성격이 다르다.

| 구분 | NFT 민팅 | ERC-20 토큰 민팅 |
| --- | --- | --- |
| 단위 | 고유한 tokenId | 수량 |
| 결과 | 특정 NFT의 소유자 기록 | 특정 주소의 잔액 증가 |
| 예시 | 컬렉션 NFT 1개 발행 | 보상 토큰 100개 발행 |
| 주로 보는 값 | owner, tokenId, metadata | balance, totalSupply |

처음에는 "NFT 민팅"과 "토큰 민팅"을 같은 단어의 다른 적용이라고 보면 된다.

## mint와 transfer는 다르다

민팅을 이해할 때 가장 많이 헷갈리는 비교가 전송이다.

이미 발행된 NFT를 다른 사람에게 보내거나 판매하는 것은 보통 transfer다. 반면 민팅은 새로 발행하거나 처음 배정하는 쪽이다.

![mint와 transfer의 차이](/images/minting-vs-transfer.png)

아주 단순하게 보면 이렇다.

```text
mint:
없음 -> 내 주소

transfer:
A 주소 -> B 주소
```

NFT 전송에서는 이미 존재하는 tokenId의 owner가 바뀐다.

```text
before:
tokenId 42의 owner = A 주소

after:
tokenId 42의 owner = B 주소
```

민팅에서는 이전 소유자가 일반 사용자 주소가 아닐 수 있다. 새 토큰이 처음 만들어지는 것이므로, 이벤트 기록에서는 `0x000...000` 같은 zero address에서 나온 것처럼 표현되는 경우도 있다.

이 표현은 "진짜 어떤 사람이 0x0 주소에서 보냈다"기보다, 새 토큰이 발행됐다는 신호로 읽는 편이 자연스럽다.

## 민팅 가격과 gas를 나눠서 보기

민팅 페이지에서 사용자가 지불하는 비용은 크게 두 종류로 나눠볼 수 있다.

| 항목 | 의미 | 누가 정하는가 |
| --- | --- | --- |
| 민팅 가격 | NFT나 토큰을 발행하기 위해 컨트랙트에 보내는 금액 | 프로젝트나 컨트랙트 규칙 |
| gas | 트랜잭션 실행을 위해 네트워크에 지불하는 비용 | 네트워크 상황과 트랜잭션 복잡도 |

예를 들어 어떤 NFT의 민팅 가격이 `0.05 ETH`라고 해보자. 사용자는 여기에 gas까지 추가로 지불해야 할 수 있다.

```text
민팅 가격: 0.05 ETH
gas 비용: 네트워크 상황에 따라 변함
```

민팅 가격은 컨트랙트가 받는 금액에 가깝다. gas는 블록체인 네트워크에서 트랜잭션을 처리하는 비용이다.

그래서 지갑 팝업에서 총 비용을 볼 때는 둘을 섞어 생각하지 않는 것이 좋다.

```text
내가 컨트랙트에 보내는 금액은 얼마인가?
네트워크 실행 비용은 얼마인가?
```

특히 인기 있는 민팅에서는 많은 사람이 동시에 트랜잭션을 보내면서 gas가 올라갈 수 있다. 민팅에 실패해도 실행 시도에 사용된 gas가 일부 소모될 수 있다는 점도 조심해야 한다.

## 왜 민팅에 실패할까

민팅 트랜잭션은 서명했다고 항상 성공하는 것이 아니다.

컨트랙트가 가진 조건을 만족하지 못하면 실패할 수 있다.

- 이미 품절됐다.
- 민팅 시간이 아직 시작되지 않았다.
- 지갑당 허용 수량을 넘었다.
- allowlist에 없는 주소다.
- 가격을 잘못 보냈다.
- 잘못된 네트워크에서 시도했다.
- 컨트랙트가 pause 상태다.

블록체인에서는 트랜잭션이 블록에 포함된 뒤에도 컨트랙트 실행 결과가 실패로 끝날 수 있다. 이 경우 상태 변경은 반영되지 않지만, 실행 시도에 든 gas는 사라질 수 있다.

그래서 민팅을 볼 때는 "버튼이 눌렸는가"보다 "트랜잭션이 성공했는가"를 확인해야 한다.

블록 탐색기에서는 보통 transaction status, event log, token transfer 기록을 같이 볼 수 있다.

## Allowlist와 public mint

NFT 민팅에서 allowlist라는 말도 자주 나온다.

allowlist는 특정 주소 목록에 있는 사람만 먼저 민팅할 수 있게 하는 방식이다. 예전에는 whitelist라는 표현도 많이 썼지만, 요즘은 allowlist라는 말을 더 자주 쓴다.

처음에는 이렇게 구분하면 된다.

| 방식 | 의미 |
| --- | --- |
| allowlist mint | 허용된 주소만 정해진 조건으로 민팅 |
| public mint | 누구나 조건을 만족하면 민팅 |
| free mint | 민팅 가격은 0이지만 gas는 들 수 있음 |

allowlist 민팅에서는 내 주소가 목록에 포함됐다는 증명이 필요할 수 있다. 컨트랙트는 그 주소가 허용된 주소인지 검증한 뒤 민팅을 허용한다.

public mint는 더 단순해 보이지만, 인기 있는 프로젝트라면 경쟁이 심해질 수 있다. 동시에 많은 사람이 민팅하면 품절, 높은 gas, 트랜잭션 실패를 볼 수 있다.

free mint라는 표현도 조심해서 봐야 한다.

```text
free mint = 민팅 가격이 0일 수 있음
gas까지 0이라는 뜻은 아님
```

## Reveal은 민팅과 별개의 단계일 수 있다

NFT 프로젝트에서는 reveal이라는 말도 자주 나온다.

어떤 프로젝트는 민팅 직후에 바로 최종 이미지와 속성을 보여준다. 반대로 어떤 프로젝트는 처음에는 같은 placeholder 이미지를 보여주고, 일정 시간이 지난 뒤 실제 이미지와 속성을 공개한다.

이때 민팅과 reveal은 다른 단계일 수 있다.

```text
민팅:
내 주소가 특정 tokenId를 소유하게 됨

reveal:
그 tokenId의 메타데이터와 이미지가 공개되거나 갱신됨
```

그래서 민팅 직후에 지갑이나 마켓플레이스에서 이미지가 바로 보이지 않는다고 해서 무조건 실패한 것은 아니다. 먼저 트랜잭션 성공 여부와 소유권 기록을 확인해야 한다.

다만 메타데이터가 어디에 저장되는지, 프로젝트가 나중에 바꿀 수 있는지, 최종 이미지가 언제 공개되는지는 프로젝트마다 다르다. NFT를 볼 때는 "소유권"과 "표시 정보"를 나눠서 읽는 습관이 필요하다.

## 민팅 전에는 무엇을 확인해야 할까

민팅은 지갑 서명을 동반하는 트랜잭션이다. 그래서 화면이 예뻐 보이거나 링크가 유명해 보여도, 지갑 팝업에서 무엇이 실행되는지 확인해야 한다.

![민팅 전 체크리스트](/images/minting-checklist.png)

처음에는 아래 정도만 확인해도 실수를 줄일 수 있다.

- 공식 사이트나 공식 채널에서 온 링크인가
- 컨트랙트 주소가 공식 정보와 같은가
- 내가 선택한 네트워크가 맞는가
- 민팅 가격과 수량이 예상과 같은가
- 지갑 팝업의 `to` 주소가 낯설지 않은가
- 단순 메시지 서명인지, 트랜잭션 서명인지 구분했는가
- `approve` 요청이 같이 뜨지는 않는가
- gas가 과하게 높거나 이상하지 않은가

특히 모르는 사이트에서 민팅하라고 하면서 지갑 서명을 요구할 때는 멈춰야 한다. 어떤 서명은 단순 로그인에 가깝지만, 어떤 요청은 자산 이동이나 권한 부여로 이어질 수 있다.

[[simple-payment-blockchain-signature|간편결제와 블록체인 서명]]을 비교할 때와 마찬가지로, 지갑에서 "확인"을 누르는 것은 가벼운 버튼 클릭이 아니다.

## 개발자가 민팅을 볼 때의 관점

개발자 관점에서는 민팅을 UI, 지갑, 컨트랙트, 인덱싱으로 나눠보면 이해하기 쉽다.

| 영역 | 보는 것 |
| --- | --- |
| UI | 사용자가 민팅 수량, 가격, 네트워크를 이해할 수 있는가 |
| 지갑 | 서명 요청이 사람이 읽기 좋게 보이는가 |
| 컨트랙트 | mint 조건, 공급량, 권한, 실패 조건이 명확한가 |
| 인덱싱 | 민팅 성공 후 이벤트와 소유권을 빠르게 반영하는가 |

사용자에게는 "민팅 완료" 한 줄로 보이지만, 실제 제품에서는 여러 경계가 만난다.

프론트엔드는 컨트랙트 주소와 함수 호출을 준비한다. 지갑은 사용자의 서명을 만든다. 컨트랙트는 규칙을 검증하고 상태를 바꾼다. 인덱서나 서버는 이벤트를 읽어 화면에 최신 결과를 보여준다.

그래서 민팅 기능을 만들거나 분석할 때는 아래 질문을 던지면 좋다.

```text
누가 민팅 요청을 만들었는가?
어떤 주소가 서명했는가?
어떤 컨트랙트 함수가 실행됐는가?
실패 조건은 무엇인가?
성공 후 어떤 이벤트와 상태가 바뀌었는가?
```

이 질문들은 NFT뿐 아니라 토큰 발행, 보상 청구, 바우처 발급 같은 온체인 발행 흐름을 볼 때도 도움이 된다.

## 처음 실습할 때 볼 것

실습은 실제 자산이 없는 테스트 환경에서 하는 편이 좋다.

1. 테스트용 지갑을 만든다.
2. 테스트넷에서 간단한 NFT 민팅 예제를 실행한다.
3. 트랜잭션 해시를 블록 탐색기에서 연다.
4. status가 success인지 확인한다.
5. event log에서 Transfer 또는 mint 관련 이벤트를 본다.
6. tokenId와 owner 주소를 확인한다.
7. 지갑이나 프론트엔드가 결과를 어떻게 갱신하는지 본다.

이 실습의 목표는 멋진 NFT를 얻는 것이 아니다. 버튼 하나 뒤에서 지갑 서명, 컨트랙트 실행, 상태 기록, 화면 갱신이 어떻게 이어지는지 확인하는 것이다.

## 참고하면 좋은 문서

더 깊게 보고 싶다면 아래 문서를 먼저 보면 좋다.

- [ERC-20 Token Standard](https://ethereum.org/developers/docs/standards/tokens/erc-20/)
- [ERC-721 Non-Fungible Token Standard](https://ethereum.org/developers/docs/standards/tokens/erc-721/)
- [ERC-1155 Multi Token Standard](https://ethereum.org/developers/docs/standards/tokens/erc-1155/)

처음부터 표준 문서의 모든 함수를 외울 필요는 없다. `mint`, `owner`, `balance`, `totalSupply`, `Transfer event`가 실제 트랜잭션과 어떻게 연결되는지 보는 것만으로도 충분하다.

## 정리

민팅은 블록체인에서 새 토큰 기록을 발행하는 동작이다.

```text
NFT 민팅은 새 tokenId의 소유권을 기록한다.
토큰 민팅은 특정 주소의 잔액과 전체 공급량을 늘릴 수 있다.
민팅은 전송과 다르다.
민팅에는 민팅 가격과 gas가 따로 있을 수 있다.
성공 여부는 블록 탐색기의 트랜잭션 상태와 이벤트로 확인한다.
```

민팅을 이해하는 목표는 "NFT를 얻는 버튼"을 외우는 것이 아니다. 지갑에서 확인을 누르는 순간, 어떤 컨트랙트가 어떤 새 기록을 만들고 그 결과가 어느 주소에 남는지 읽을 수 있게 되는 것이다.
