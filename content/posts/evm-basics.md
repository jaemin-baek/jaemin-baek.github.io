![EVM 이해하기](/images/evm-basics-cover.png)

[[web3-service-basics|Web3 기반 서비스]]를 공부하다 보면 어느 순간 EVM이라는 단어가 나온다. 이더리움 자료에도 나오고, Polygon, Arbitrum, BNB Chain 같은 체인을 볼 때도 "EVM 호환"이라는 표현이 따라붙는다.

처음에는 이렇게 느껴질 수 있다.

```text
EVM은 이더리움 네트워크 자체인가?
스마트 컨트랙트는 EVM 안에 들어 있는 건가?
지갑이 트랜잭션에 서명하면 EVM이 바로 실행하는 건가?
```

나도 처음에는 EVM을 너무 큰 단어로 느꼈다. 하지만 지갑, 트랜잭션, 스마트 컨트랙트 흐름 위에 올려보면 생각보다 단순하게 잡을 수 있다.

이 글은 EVM 내부 opcode를 깊게 파고드는 글은 아니다. [[eoa-externally-owned-account|EOA]], [[simple-payment-blockchain-signature|블록체인 서명]], [[token-basics|토큰]] 글에서 봤던 흐름을 바탕으로, "트랜잭션이 들어오면 스마트 컨트랙트 코드는 어디서 어떻게 실행되는가?"를 이해하기 위한 메모다.

## 먼저 한 문장으로 보기

EVM을 한 문장으로 줄이면 이렇게 볼 수 있다.

```text
EVM은 이더리움 계열 체인에서 스마트 컨트랙트 코드를 실행하고 상태 변경을 계산하는 공통 실행 환경이다.
```

여기서 중요한 단어는 세 개다.

- 스마트 컨트랙트 코드
- 실행
- 상태 변경

EVM은 지갑 앱이 아니다. 블록 탐색기도 아니다. 특정 서버 한 대도 아니다.

각 노드가 트랜잭션을 검증하고 블록을 처리할 때, 스마트 컨트랙트 코드를 같은 규칙으로 실행하기 위해 사용하는 가상의 실행 환경에 가깝다.

```text
같은 입력
같은 코드
같은 상태
같은 EVM 규칙

=> 같은 결과
```

블록체인 네트워크에서 이 성질은 중요하다. 여러 노드가 각자 계산해도 결과가 같아야 같은 장부를 공유할 수 있기 때문이다.

## EVM은 체인 자체가 아니다

EVM을 처음 볼 때 가장 먼저 나눠야 하는 것은 "네트워크"와 "실행 환경"이다.

Ethereum 네트워크에는 노드들이 있다. 노드들은 트랜잭션과 블록을 받고, 규칙에 맞는지 검증하고, 상태를 업데이트한다.

그 과정에서 스마트 컨트랙트 코드 실행이 필요하면 EVM 규칙에 따라 실행한다.

```text
Ethereum 네트워크
= 노드들이 연결된 전체 시스템

EVM
= 노드가 스마트 컨트랙트 코드를 실행할 때 쓰는 공통 실행 환경
```

비유하자면 EVM은 "블록체인 안의 작은 컴퓨터"처럼 설명되곤 한다. 완전히 틀린 말은 아니지만, 실제 컴퓨터 한 대가 어딘가에 따로 있는 것은 아니다.

더 정확히는 모든 노드가 같은 방식으로 실행할 수 있도록 정의된 가상 머신 규칙에 가깝다.

## 전체 흐름에서 EVM 위치 보기

[[web3-service-basics|Web3 기반 서비스]]에서는 사용자의 의도가 트랜잭션이 되고, 지갑 서명을 거쳐 네트워크에서 검증된다고 봤다.

그 흐름 안에서 EVM은 주로 스마트 컨트랙트 실행 단계에 있다.

![EVM 실행 흐름](/images/evm-execution-flow-handdrawn.png)

단순화하면 이런 순서다.

```text
1. dApp이 트랜잭션 요청을 만든다.
2. 지갑이 사용자에게 내용을 보여준다.
3. 사용자가 승인하면 EOA의 개인키로 서명한다.
4. 서명된 트랜잭션이 네트워크로 전파된다.
5. 노드가 서명, nonce, gas, 잔액 같은 조건을 검증한다.
6. 컨트랙트 코드 실행이 필요하면 EVM이 실행 결과를 계산한다.
7. 결과가 블록체인 상태에 반영된다.
```

여기서 지갑은 EVM을 직접 "돌리는" 앱이라기보다, 사용자의 승인과 서명을 만드는 쪽에 가깝다. EVM 실행은 네트워크의 노드들이 블록을 처리하면서 수행한다.

물론 지갑이나 dApp이 실행 결과를 미리 예측하기 위해 `eth_call` 같은 방식으로 노드에 시뮬레이션을 요청할 수는 있다. 하지만 실제 상태 변경은 트랜잭션이 블록에 포함되고 노드들이 그 결과를 받아들일 때 일어난다.

## 트랜잭션은 EVM에 무엇을 줄까

트랜잭션은 단순히 "보내기 버튼을 눌렀다"가 아니다. EVM 관점에서는 실행에 필요한 입력이 담긴 요청이다.

![트랜잭션이 EVM에 주는 것](/images/evm-transaction-inputs-handdrawn.png)

대표적으로 이런 값들이 중요하다.

| 값 | 의미 |
| --- | --- |
| `from` | 이 트랜잭션을 서명한 계정 |
| `to` | 받을 주소 또는 호출할 컨트랙트 주소 |
| `value` | 함께 보낼 네이티브 코인 수량 |
| `data` | 컨트랙트 함수 호출에 필요한 입력 데이터 |
| `nonce` | 같은 계정에서 보낸 트랜잭션 순서 |
| `gas` | 실행에 사용할 수 있는 최대 연료 |
| `chainId` | 어떤 체인에서 유효한 서명인지 구분하는 값 |

단순 ETH 전송이라면 `to`와 `value`가 핵심일 수 있다. 반면 ERC-20 토큰 전송이나 swap처럼 컨트랙트를 호출하는 행동은 `data`가 중요해진다.

예를 들어 [[token-basics|토큰 전송]]에서 `transfer(B, 10)`을 호출한다고 해보자. 사용자는 지갑 화면에서 "B에게 10개 전송"처럼 볼 수 있지만, 트랜잭션 안에는 컨트랙트가 이해할 수 있는 형태의 `data`가 들어간다.

```text
사용자가 보는 의미:
B에게 ABC 10개 전송

트랜잭션이 담는 실행 입력:
ABC 컨트랙트 주소
transfer 함수 식별자
받는 주소 B
수량 10
```

EVM은 이 입력과 현재 상태를 가지고 컨트랙트 코드를 실행한다.

## Solidity가 바로 실행되는 것은 아니다

스마트 컨트랙트를 볼 때 Solidity 코드를 많이 본다. 그래서 EVM이 Solidity를 직접 읽고 실행한다고 생각하기 쉽다.

하지만 보통 흐름은 이렇다.

```text
Solidity 코드
-> 컴파일
-> EVM bytecode
-> 배포 트랜잭션
-> 컨트랙트 주소에 코드 저장
-> 이후 트랜잭션에서 bytecode 실행
```

즉 EVM이 실행하는 것은 사람이 읽기 좋은 Solidity 원문이 아니라, 컴파일된 bytecode다.

개발자는 Solidity 같은 언어로 컨트랙트를 작성한다. 컴파일러는 그 코드를 EVM이 이해할 수 있는 bytecode로 바꾼다. 컨트랙트가 배포되면 그 bytecode가 체인 상태의 일부로 저장된다.

그 다음 누군가 컨트랙트 함수를 호출하는 트랜잭션을 보내면, EVM은 저장된 bytecode를 실행한다.

```text
소스 코드:
사람과 개발 도구가 읽기 좋다.

bytecode:
EVM이 실행한다.

ABI:
dApp과 지갑이 함수 호출 데이터를 만들고 해석할 때 도움을 준다.
```

여기서 ABI도 자주 나온다. ABI는 컨트랙트 함수 이름, 파라미터, 반환값을 외부 앱이 이해할 수 있게 설명하는 인터페이스 정보에 가깝다. dApp은 ABI를 보고 "이 함수 호출을 어떤 `data`로 인코딩해야 하는지" 알 수 있다.

## Gas는 EVM 실행 연료다

EVM을 이해할 때 gas는 거의 항상 따라온다.

gas는 단순 수수료 이름이 아니라, EVM 실행에 필요한 계산 자원을 재는 단위다.

스마트 컨트랙트 코드는 반복문을 돌 수도 있고, 저장소를 읽고 쓸 수도 있고, 다른 컨트랙트를 호출할 수도 있다. 이런 실행을 무제한으로 허용하면 네트워크가 멈출 수 있다.

그래서 EVM은 각 연산에 비용을 둔다.

```text
연산 실행
-> 정해진 gas 차감
-> gas가 부족하면 실행 실패
```

사용자는 트랜잭션을 보낼 때 어느 정도의 gas를 쓸 수 있는지 한도를 둔다. 그리고 실제로 사용된 gas에 gas price 계열 값을 곱해 수수료를 낸다.

[[eoa-externally-owned-account|EOA]] 글에서 봤던 것처럼 EIP-1559 이후 트랜잭션에서는 `maxFeePerGas`, `maxPriorityFeePerGas` 같은 값도 같이 본다.

처음에는 이렇게만 잡아도 충분하다.

```text
gas limit:
이 트랜잭션이 최대 얼마만큼 실행될 수 있는가

gas price 계열 값:
gas 한 단위에 얼마를 낼 것인가

사용된 gas:
실제로 EVM 실행과 상태 처리에 소비된 양
```

중요한 점은 컨트랙트 실행이 실패해도 gas가 일부 또는 대부분 사용될 수 있다는 것이다. EVM이 실행을 시도했고, 그 과정에서 계산 자원을 썼기 때문이다.

그래서 지갑이나 dApp은 트랜잭션을 보내기 전에 예상 gas를 보여주고, 실패 가능성이 큰 요청은 사용자에게 더 조심스럽게 보여줘야 한다.

## EVM이 바꾸는 것은 상태다

EVM 실행의 결과는 결국 상태 변경으로 이어진다.

여기서 상태는 대략 이런 것들을 포함한다.

| 상태 | 예시 |
| --- | --- |
| 계정 잔액 | 어떤 주소가 가진 ETH 수량 |
| 컨트랙트 코드 | 특정 컨트랙트 주소에 저장된 bytecode |
| 컨트랙트 storage | ERC-20 잔액, allowance, NFT 소유자 기록 |
| nonce | 계정이 보낸 트랜잭션 순서 |

예를 들어 ERC-20 `transfer`가 성공하면 내 지갑 앱 안의 토큰 파일이 이동하는 것이 아니다.

토큰 컨트랙트의 storage가 바뀐다.

```text
balances[A] = 100
balances[B] = 20

transfer(B, 10) 실행 후

balances[A] = 90
balances[B] = 30
```

[[swap-token-exchange|swap]]도 마찬가지다. 사용자가 버튼 하나를 눌렀지만, 실제로는 router와 pool 컨트랙트의 storage, 여러 토큰 컨트랙트의 잔액 기록, 이벤트 로그가 함께 움직일 수 있다.

그래서 EVM을 볼 때는 "코드가 실행됐다"에서 멈추지 말고, 이 질문까지 이어가야 한다.

```text
실행 결과 어떤 상태가 바뀌었는가?
```

## Event log는 storage와 다르다

블록 탐색기에서 트랜잭션을 보면 logs나 events를 자주 본다.

이벤트 로그는 컨트랙트가 실행 중에 남기는 기록이다. 지갑, 탐색기, 백엔드 인덱서가 트랜잭션 결과를 해석할 때 많이 사용한다.

하지만 이벤트 로그와 storage는 다르다.

```text
storage:
컨트랙트가 이후 실행에서도 읽을 수 있는 상태

event log:
트랜잭션 실행 중 발생한 기록
```

ERC-20 전송에서 `Transfer` 이벤트가 보인다고 해서, 그 이벤트 자체가 잔액을 들고 있는 것은 아니다. 잔액은 컨트랙트 storage에 있고, 이벤트는 "이런 전송이 일어났다"는 기록에 가깝다.

블록 탐색기가 token transfers를 보기 좋게 보여줄 수 있는 이유도 이런 이벤트 로그를 읽고 해석하기 때문이다.

## EVM 호환 체인은 무엇일까

"EVM 호환"이라는 말도 자주 나온다.

처음에는 이렇게 이해하면 된다.

```text
EVM 호환 체인
= EVM 실행 모델과 Ethereum 계열 주소, 트랜잭션, 스마트 컨트랙트 개발 경험을 상당 부분 공유하는 체인
```

즉 Ethereum에서 쓰던 Solidity 컨트랙트, 지갑 주소 형식, ABI, RPC, 개발 도구를 비슷하게 사용할 수 있는 경우가 많다.

그래서 하나의 지갑 앱에서 여러 EVM 계열 네트워크를 지원할 수 있다.

하지만 "호환"이 "완전히 같은 체인"이라는 뜻은 아니다.

| 구분 | 달라질 수 있는 것 |
| --- | --- |
| `chainId` | 어떤 네트워크인지 구분 |
| native coin | gas로 쓰는 기본 자산 |
| RPC endpoint | 트랜잭션을 보낼 노드 주소 |
| 수수료 정책 | gas price, L2 수수료 구조 |
| 컨트랙트 주소 | 같은 토큰 이름이라도 체인마다 다를 수 있음 |
| 블록 생성 방식 | 합의 구조와 처리 속도 |

그래서 지갑 개발에서는 chainId를 아주 중요하게 봐야 한다. 같은 `0x...` 주소처럼 보여도, 어떤 체인에서 어떤 컨트랙트와 상호작용하는지에 따라 완전히 다른 행동이 된다.

[[ticker-token-symbol|토큰 티커]]만 믿으면 안 되는 이유도 여기에 있다. 같은 심볼의 토큰이 여러 체인에 있을 수 있고, 컨트랙트 주소도 다를 수 있다.

## 지갑 개발자 관점에서 보기

EVM 개념은 지갑 앱을 만들 때 바로 실무 흐름으로 이어진다.

지갑은 EVM을 직접 구현하지 않더라도, EVM 트랜잭션이 무엇을 의미하는지 사용자에게 보여줘야 한다.

예를 들어 지갑 SDK나 wallet-core에서는 이런 일을 나눠서 볼 수 있다.

```text
계정:
니모닉, seed, BIP-44 path, private key, address

서명:
메시지 서명, typed data 서명, 트랜잭션 서명

트랜잭션:
to, value, data, nonce, gas, chainId 구성

네트워크:
RPC 요청, gas 추정, nonce 조회, raw transaction 전파, receipt 확인

UX:
사용자가 무엇에 서명하는지 사람이 읽을 수 있게 표시
```

여기서 "EVM 기반 서명"은 단순히 `sign()` 함수를 호출했다는 뜻으로 끝나지 않는다.

사용자가 서명하려는 것이 메시지인지, typed data인지, 실제 상태 변경 트랜잭션인지 구분해야 한다. `chainId`, `to`, `value`, `data`, gas 정보를 확인해야 하고, 컨트랙트 호출이라면 가능한 범위에서 함수 의미를 해석해 보여줘야 한다.

특히 WalletConnect 같은 흐름에서는 외부 dApp이 지갑에 요청을 보낸다.

```text
dApp 요청
-> 지갑에서 요청 해석
-> 사용자에게 표시
-> 인증/PIN/생체 확인
-> 개인키로 서명
-> 서명값 또는 raw transaction 반환
```

이때 지갑은 단순 전달자가 아니다. 사용자가 위험한 요청을 눈치챌 수 있도록 네트워크, 주소, 권한, 수량, gas, 컨트랙트 호출 정보를 최대한 명확하게 보여줘야 한다.

[[android-wallet-secret-storage-keystore-tink|Android Keystore와 Tink AEAD로 개인키 저장하기]]에서 봤던 개인키 저장 구조도 여기로 이어진다. 개인키는 안전하게 보관하고, 서명 시점에만 짧게 꺼내 사용한 뒤, 밖으로는 서명값이나 signed transaction만 나가야 한다.

## 자주 헷갈리는 것들

EVM을 처음 볼 때는 아래 구분이 특히 자주 헷갈린다.

| 헷갈리는 말 | 이렇게 나눠보기 |
| --- | --- |
| EVM = Ethereum? | Ethereum은 네트워크, EVM은 실행 환경 |
| EVM = Solidity? | Solidity는 소스 언어, EVM은 bytecode 실행 환경 |
| 지갑이 EVM을 실행한다? | 지갑은 서명하고, 노드가 실행과 검증을 담당 |
| 토큰이 지갑 안에 있다? | 토큰 잔액은 보통 컨트랙트 storage에 있음 |
| 실패한 트랜잭션은 비용이 없다? | 실행을 시도한 gas는 소모될 수 있음 |
| 이벤트가 상태다? | 이벤트는 로그, storage와 다름 |

이 표를 조금 더 짧게 줄이면 이렇다.

```text
지갑은 서명한다.
노드는 검증한다.
EVM은 컨트랙트 코드를 실행한다.
컨트랙트 storage가 상태를 가진다.
블록체인은 그 결과를 공유한다.
```

## 처음 실습할 때 볼 것

EVM은 글로만 보면 추상적이다. 실제 트랜잭션 하나를 블록 탐색기에서 열어보는 편이 훨씬 빠르다.

처음에는 아래 항목만 봐도 충분하다.

1. `From`은 어떤 EOA인가?
2. `To`는 EOA인가, 컨트랙트인가?
3. `Value`가 있는가?
4. `Input Data`가 비어 있는가, 함수 호출 데이터가 있는가?
5. `Gas Used`는 어느 정도인가?
6. `Logs`에는 어떤 이벤트가 남았는가?
7. Token Transfers가 있다면 어떤 컨트랙트 이벤트에서 나온 것인가?

특히 `Input Data`가 있는 트랜잭션을 보면 "EVM이 실행할 입력"이라는 감각이 잡힌다. `To`가 ERC-20 컨트랙트이고 `Input Data`가 `transfer` 호출이라면, 그 트랜잭션은 단순 코인 전송이 아니라 컨트랙트 함수를 실행해 storage를 바꾸는 요청이다.

## 정리

EVM은 이더리움 계열 스마트 컨트랙트 실행을 이해하는 중심 개념이다.

```text
트랜잭션은 실행 요청을 담는다.
지갑은 사용자의 개인키로 요청에 서명한다.
노드는 서명, nonce, gas, 잔액을 검증한다.
EVM은 컨트랙트 bytecode를 실행한다.
실행 결과는 storage, logs, gas 사용량, 성공/실패로 남는다.
그 결과가 블록체인 상태에 반영된다.
```

EVM을 이해한다는 것은 가상 머신의 모든 내부 명령어를 외우는 일이 아니다.

사용자가 지갑에서 "확인"을 누르는 순간, 어떤 트랜잭션이 어떤 컨트랙트 코드를 실행하고, 어떤 상태를 바꾸며, 그 비용과 위험이 어디에서 생기는지 읽을 수 있게 되는 것이다.
