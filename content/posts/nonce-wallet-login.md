![Nonce 발급과 지갑 로그인](/images/nonce-wallet-login-cover.png)

[[web3-service-basics|Web3 기반 서비스]]를 볼 때는 보통 이런 흐름을 먼저 잡았다.

```text
서비스는 무엇을 요청하고,
지갑은 무엇을 승인하고,
노드는 무엇을 검증하고,
컨트랙트는 무엇을 바꾸는가?
```

그런데 지갑 로그인으로 들어오면 조금 다른 질문이 생긴다.

```text
로그인인데 왜 nonce를 발급하지?
지갑으로 서명한다는데, 이게 트랜잭션인가?
nonce는 블록체인에서 쓰는 순서 번호 아닌가?
```

처음에는 이 부분이 꽤 헷갈린다. 블록체인 글에서 본 nonce는 트랜잭션 순서 번호였는데, API 문서에서는 `/nonce`를 호출해서 서버가 nonce를 발급한다고 말하기 때문이다.

이 글은 nonce를 암호학적으로 깊게 설명하는 글은 아니다. Web3 서비스에서 지갑 로그인을 구현하거나 읽을 때 필요한 정도로, "서버가 왜 일회용 값을 만들고, 지갑은 무엇에 서명하며, 서버는 무엇을 검증하는가"를 정리한 메모다.

## 먼저 한 문장으로 보기

지갑 로그인에서 nonce 발급을 한 문장으로 줄이면 이렇게 볼 수 있다.

```text
Nonce 발급은 서버가 지갑 소유자 확인을 위해 일회용 질문을 만드는 단계다.
```

여기서 중요한 단어는 세 개다.

- 서버
- 일회용
- 질문

사용자는 비밀번호를 입력하는 대신 지갑으로 메시지에 서명한다. 서버는 그 서명을 보고 "이 주소를 제어하는 지갑이 맞구나"라고 확인한다.

이때 서버가 매번 새로운 nonce를 넣어주지 않으면, 과거에 만든 서명을 누군가 다시 제출할 수 있다. 그래서 nonce는 로그인 요청마다 새로 발급되고, 한 번 사용한 뒤에는 폐기되어야 한다.

```text
nonce = 비밀번호가 아니라, 한 번만 쓰는 인증 질문
```

## 전체 흐름

지갑 로그인에서 nonce가 들어가는 흐름은 대략 이렇게 볼 수 있다.

![Nonce 로그인 흐름](/images/nonce-login-flow-handdrawn.png)

단계를 조금 풀면 이렇다.

1. 사용자가 서비스에서 지갑 로그인을 누른다.
2. 서비스 서버가 랜덤한 nonce를 만든다.
3. 서버는 nonce를 주소, 만료 시간, 사용 여부와 함께 저장한다.
4. 서비스는 nonce가 포함된 로그인 메시지를 지갑에 보여준다.
5. 사용자가 지갑에서 메시지를 확인하고 서명한다.
6. 서버는 메시지와 서명을 받아 검증한다.
7. 검증에 성공하면 nonce를 사용 완료로 바꾸고 세션을 만든다.

이 흐름에서 체인 상태는 보통 바뀌지 않는다. 메시지 서명은 "내가 이 주소를 제어한다"는 증명에 가깝고, 토큰 전송이나 컨트랙트 실행 같은 온체인 트랜잭션과는 다르다.

[[simple-payment-blockchain-signature|간편결제와 블록체인 서명]]에서 나눴던 표현을 빌리면, 여기서의 서명은 보통 메시지 서명이다.

```text
메시지 서명:
체인 상태 변경 없음
보통 gas 없음
서버가 서명을 검증해 로그인 세션 생성

트랜잭션 서명:
체인 상태 변경 있음
보통 gas 있음
노드가 트랜잭션을 검증하고 블록에 포함
```

그래서 지갑 로그인 nonce를 볼 때는 "이 트랜잭션이 몇 번째인가"보다 "이 로그인 요청이 이번에 새로 만들어진 요청인가"를 먼저 봐야 한다.

## 왜 nonce가 필요할까

nonce가 없다고 가정해보자.

서비스가 아래 메시지만 지갑에 서명하게 한다.

```text
Login to My DApp
Address: 0xABCD...
```

사용자가 이 메시지에 한 번 서명하면, 서명값은 항상 같은 의미를 가진다. 누군가 이 서명값을 훔치거나 로그에서 가져오면 나중에 다시 서버에 보낼 수 있다.

서버 입장에서는 곤란하다.

```text
이 서명이 지금 사용자가 새로 만든 로그인 요청인지,
예전에 만들어진 서명을 누가 다시 보낸 것인지
구분하기 어렵다.
```

이런 재사용 공격을 replay attack이라고 부른다.

nonce는 이 문제를 줄이기 위해 들어간다. 서버는 매번 다른 값을 넣어 메시지를 만든다.

```text
Login to My DApp
Address: 0xABCD...
Nonce: a7f3...9c2b
Issued At: 2026-05-28T12:00:00Z
```

사용자는 이 메시지 전체에 서명한다. 그러면 서명은 `a7f3...9c2b`라는 nonce가 포함된 요청에만 유효하다.

서버는 검증할 때 이렇게 본다.

```text
이 nonce를 우리가 발급했는가?
아직 만료되지 않았는가?
아직 사용되지 않았는가?
서명에서 복구한 주소가 요청 주소와 같은가?
메시지의 domain, uri, chainId가 기대한 값과 같은가?
```

모두 맞으면 로그인시킨다. 그리고 바로 nonce를 사용 완료로 바꾼다.

```text
발급됨 -> 서명됨 -> 검증됨 -> 사용 완료
```

이후 같은 nonce와 같은 서명을 다시 보내면 서버는 거절해야 한다.

## Nonce는 비밀값일까

nonce는 비밀번호나 개인키처럼 오래 숨겨야 하는 값은 아니다. 지갑에 보여주고, 메시지에 포함하고, 서버로 다시 보내야 하기 때문이다.

하지만 아무렇게나 만들어도 되는 값은 아니다.

좋은 nonce는 보통 이런 성질을 가져야 한다.

- 예측하기 어렵다.
- 충분히 길다.
- 같은 값이 다시 나오기 어렵다.
- 서버에 저장되어 검증할 수 있다.
- 짧은 만료 시간이 있다.
- 한 번 사용하면 다시 쓸 수 없다.

즉 nonce는 "공개돼도 되는 값"에 가깝지만, "공격자가 미리 맞히기 쉬운 값"이면 안 된다.

나쁜 예시는 이런 쪽이다.

```text
nonce = "1234"
nonce = 현재 분 단위 시간
nonce = address 뒤 4자리
nonce = 1씩 증가하는 로그인 번호
```

이런 값은 공격자가 예측하기 쉽다. 로그인 nonce는 서버에서 안전한 난수 생성기로 만들고, DB나 캐시 같은 서버 저장소에 기록해두는 편이 좋다.

## 메시지에는 nonce만 있으면 될까

nonce만 넣어도 replay 방지에는 도움이 된다. 하지만 실제 로그인 메시지는 보통 nonce 말고도 여러 정보를 같이 넣는다.

대표적으로는 이런 값들이 들어간다.

| 필드 | 이유 |
| --- | --- |
| domain | 어떤 서비스가 서명을 요청했는지 보여준다 |
| address | 어떤 지갑 주소로 로그인하는지 고정한다 |
| uri | 어떤 서비스 URL에 대한 로그인인지 묶는다 |
| chainId | 어떤 네트워크 기준의 계정인지 나타낸다 |
| nonce | 재사용을 막는 일회용 값이다 |
| issuedAt | 메시지가 언제 만들어졌는지 나타낸다 |
| expirationTime | 언제까지 유효한지 제한한다 |

이런 형식을 표준화한 흐름으로는 Sign-In with Ethereum, 줄여서 SIWE가 있다. SIWE는 Ethereum 계정이 오프체인 서비스에 로그인할 때 사용할 수 있는 메시지 형식을 정의한다.

예시는 이런 모양에 가깝다.

```text
example.com wants you to sign in with your Ethereum account:
0xABCD...

URI: https://example.com/login
Version: 1
Chain ID: 1
Nonce: a7f39c2b
Issued At: 2026-05-28T12:00:00Z
Expiration Time: 2026-05-28T12:10:00Z
```

처음부터 모든 필드를 외울 필요는 없다. 다만 메시지가 너무 짧거나, 어떤 서비스에 로그인하는지 알 수 없거나, nonce와 만료 시간이 없다면 조심해서 봐야 한다.

```text
좋은 로그인 메시지는 사용자가 무엇에 서명하는지 읽을 수 있어야 한다.
```

## 서버는 무엇을 검증할까

서버 검증은 단순히 "서명값이 있다"를 보는 일이 아니다.

대략 아래 순서로 봐야 한다.

```text
1. 메시지 형식이 기대한 형식인가?
2. domain과 uri가 우리 서비스인가?
3. nonce가 서버에서 발급한 값인가?
4. nonce가 아직 만료되지 않았는가?
5. nonce가 아직 사용되지 않았는가?
6. 서명에서 복구한 주소가 메시지의 address와 같은가?
7. 필요한 경우 chainId가 허용된 네트워크인가?
8. 모두 맞으면 nonce를 사용 완료로 바꾸고 세션을 만든다.
```

여기서 중요한 구현 포인트는 마지막에 있다. nonce를 사용 완료로 바꾸는 처리는 가능하면 원자적으로 해야 한다.

예를 들어 같은 서명이 거의 동시에 두 번 들어왔다고 해보자.

```text
요청 A: nonce 미사용 확인
요청 B: nonce 미사용 확인
요청 A: 로그인 성공
요청 B: 로그인 성공
```

이렇게 되면 "한 번만 쓰는 값"이라는 성질이 약해진다. 그래서 DB 업데이트 조건에 `used = false`를 포함하거나, 트랜잭션/락/원자적 consume 처리를 써서 하나만 성공하게 만들어야 한다.

```text
미사용 nonce를 확인한다
-> 같은 동작 안에서 사용 완료로 바꾼다
-> 성공한 요청만 세션을 만든다
```

작은 차이처럼 보이지만 인증 흐름에서는 꽤 중요하다.

## 트랜잭션 nonce와 로그인 nonce는 다르다

가장 헷갈리는 부분은 이름이 같다는 점이다.

[[eoa-externally-owned-account|EOA]] 글에서 본 nonce는 트랜잭션 nonce였다. 해당 EOA가 보낸 트랜잭션 순서를 나타내고, 이미 처리된 번호의 트랜잭션이 다시 실행되는 것을 막는다.

반면 지갑 로그인 nonce는 서비스 서버가 발급하는 오프체인 인증용 값이다.

![두 가지 Nonce 구분하기](/images/nonce-types-comparison-handdrawn.png)

둘을 표로 나누면 이렇다.

| 구분 | 로그인 nonce | 트랜잭션 nonce |
| --- | --- | --- |
| 위치 | 서비스 서버 | 블록체인 계정 상태 |
| 목적 | 로그인 서명 재사용 방지 | 트랜잭션 순서 관리와 중복 실행 방지 |
| 누가 만든다 | 보통 백엔드 서버 | 계정의 현재 nonce를 기반으로 지갑이나 클라이언트가 설정 |
| 검증 주체 | 서비스 서버 | 노드와 네트워크 |
| 체인 기록 | 보통 없음 | 트랜잭션에 포함되어 블록에 기록 |
| gas | 보통 없음 | 보통 있음 |

그래서 API 문서에서 "nonce 발급"이라고 나오면 보통은 로그인 nonce를 말한다.

블록 탐색기에서 트랜잭션 상세에 `Nonce: 7`처럼 보이면 트랜잭션 nonce를 말한다.

```text
로그인 nonce:
이번 로그인 요청을 위한 1회용 질문

트랜잭션 nonce:
이 EOA가 보내는 트랜잭션 순서 번호
```

같은 이름이지만 사용 위치가 다르다.

## 지갑 로그인은 온체인 로그인이 아니다

지갑으로 로그인한다고 해서 매번 블록체인에 로그인 기록이 남는 것은 아니다.

일반적인 메시지 서명 로그인은 오프체인 인증이다.

```text
체인에 기록되는 것:
없을 수 있음

서비스 서버에 남는 것:
nonce 발급 기록
서명 검증 결과
로그인 세션
사용자 주소
```

그래서 사용자는 gas를 내지 않는다. 트랜잭션이 블록에 포함될 때까지 기다리지 않아도 된다.

다만 예외가 있다. 스마트 컨트랙트 지갑은 EOA처럼 단순히 공개키 복구만으로 검증하지 못할 수 있다. 이 경우에는 ERC-1271 같은 방식으로 컨트랙트가 "이 서명이 유효하다"고 말하는지 확인해야 한다. 이때는 특정 chainId의 컨트랙트 상태를 읽어야 할 수 있다.

그래도 핵심은 같다.

```text
로그인 nonce는 서비스 세션을 만들기 위한 인증 재료다.
토큰 전송이나 컨트랙트 실행 자체는 아니다.
```

## 개발할 때 자주 놓치는 부분

지갑 로그인 nonce를 구현할 때는 아래 항목을 자주 확인하게 된다.

- nonce는 안전한 난수로 만든다.
- nonce는 서버에 저장한다.
- nonce에는 만료 시간을 둔다.
- nonce는 한 번 사용하면 폐기한다.
- 메시지에는 domain, uri, address, nonce, issuedAt을 포함한다.
- 서명 검증 후 복구된 주소와 요청 주소를 비교한다.
- 다른 도메인에서 만든 메시지를 받아들이지 않는다.
- 같은 nonce가 동시에 두 번 성공하지 않게 한다.
- 로그인 성공 후에는 nonce가 아니라 세션 토큰이나 쿠키로 사용자를 식별한다.

여기서 특히 중요한 것은 domain이다. 사용자는 지갑 화면에서 어떤 사이트가 서명을 요청하는지 볼 수 있어야 한다.

피싱 사이트가 비슷한 문구를 보여주고 서명을 받아가도, 서버가 domain과 uri를 엄격히 확인하면 다른 서비스에서 만든 메시지를 그대로 재사용하기 어렵다.

물론 사용자가 피싱 사이트에서 직접 서명하면 위험은 여전히 남는다. 그래서 지갑 화면에는 사람이 읽을 수 있는 메시지가 중요하고, 서비스도 불필요하게 애매한 문구를 만들지 않는 편이 좋다.

## 처음에는 이렇게 외우기

처음 공부할 때는 아래 세 문장만 잡아도 충분하다.

```text
Nonce 발급은 서버가 일회용 로그인 질문을 만드는 일이다.

지갑은 그 질문이 포함된 메시지에 서명한다.

서버는 서명과 nonce를 검증하고, 성공하면 nonce를 폐기한 뒤 세션을 만든다.
```

그리고 트랜잭션 nonce와는 이렇게 나눠서 보면 된다.

```text
로그인 nonce는 서비스 서버의 인증 흐름에 있다.

트랜잭션 nonce는 블록체인 계정의 트랜잭션 순서에 있다.
```

둘 다 재사용을 막는다는 공통점은 있다. 하지만 하나는 오프체인 로그인 세션을 위한 값이고, 다른 하나는 온체인 트랜잭션 처리를 위한 값이다.

이 차이를 잡으면 Web3 서비스의 로그인 API도 조금 덜 낯설어진다.

```text
서비스가 nonce를 발급한다.
지갑이 메시지에 서명한다.
서버가 서명을 검증한다.
로그인 세션이 만들어진다.
체인 상태는 보통 바뀌지 않는다.
```

여기까지 보면 지갑 로그인은 "블록체인으로 비밀번호를 대체하는 마법"이라기보다, 사용자의 주소 제어권을 서명으로 증명하고 서버 세션으로 연결하는 인증 흐름에 가깝다.
