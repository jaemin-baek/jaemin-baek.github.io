![Web3 기반 서비스란?](/images/web3-service-cover.png)

지갑을 개발하다 보면 `Web3 기반`, `Web3 서비스`, `dApp 연동` 같은 말을 자주 듣는다. 그런데 막상 "Web3 기반이 정확히 뭐예요?"라고 물으면 대답이 애매해질 때가 있다.

단어가 어려워서라기보다, 너무 많은 것을 한 번에 가리키기 때문이다.

Web3는 블록체인, 지갑, 토큰, NFT, 스마트 컨트랙트, 탈중앙화, 소유권, 커뮤니티, DAO 같은 단어와 같이 움직인다. 그래서 처음에는 "아무튼 블록체인 들어가면 Web3인가?"처럼 느껴질 수 있다.

이 글은 거창한 철학보다는, 개발자가 서비스를 만들 때 만나는 구조를 기준으로 Web3 기반 서비스를 정리하는 메모다. [[blockchain-study-guide|블록체인 학습방법 가이드]]에서 봤던 지갑, 서명, 트랜잭션 흐름을 서비스 관점으로 조금 옮겨본다.

## 먼저 한 문장으로 보기

Web3 기반 서비스를 한 문장으로 줄이면 이렇게 볼 수 있다.

```text
Web3 기반 서비스는 사용자의 지갑과 서명을 통해 블록체인 상태를 읽고 쓰는 서비스다.
```

여기서 중요한 단어는 네 개다.

- 지갑
- 서명
- 블록체인 상태
- 읽기와 쓰기

조금 더 제품 문장처럼 바꾸면 이렇게 말할 수도 있다.

```text
사용자가 블록체인 계정으로 접속하고,
자신의 키로 행동을 승인하며,
그 결과가 온체인 상태에 반영되는 서비스
```

처음에는 이 정도로 잡아도 충분하다.

Web3 기반이라는 말은 단순히 화면에 코인 가격이 보인다는 뜻이 아니다. 사용자의 지갑 주소가 계정처럼 쓰이고, 중요한 행동은 사용자의 서명을 거쳐 실행되며, 토큰이나 NFT 같은 자산 상태가 블록체인 위에 기록될 수 있다는 뜻에 가깝다.

## Web2 서비스와 무엇이 다를까

일반적인 Web2 서비스에서는 사용자가 아이디와 비밀번호로 로그인한다. 서버는 사용자 정보를 DB에 저장하고, 사용자의 행동도 대부분 서버가 처리한다.

예를 들어 쇼핑몰에서 주문 버튼을 누르면 서버가 주문 데이터를 만들고, DB에 저장하고, 결제 시스템과 통신한다. 사용자는 서버가 관리하는 계정 안에서 행동한다.

Web3 서비스에서는 여기에 지갑과 블록체인이 들어온다.

| 구분 | Web2 서비스 | Web3 기반 서비스 |
| --- | --- | --- |
| 계정 | 이메일, 아이디, 소셜 로그인 | 지갑 주소, 온체인 계정 |
| 인증 | 비밀번호, 세션, 토큰 | 메시지 서명, 지갑 연결 |
| 주요 상태 | 서비스 DB | 블록체인 상태 + 서비스 DB |
| 중요한 행동 | 서버 API 호출 | 트랜잭션 서명과 컨트랙트 실행 |
| 자산 | 서비스 내부 데이터 | 토큰, NFT, 온체인 기록 |
| 신뢰 기준 | 서비스 운영자 | 코드, 체인 상태, 사용자 서명 |

물론 현실의 Web3 서비스가 서버를 전혀 쓰지 않는다는 뜻은 아니다. 프론트엔드, 백엔드, 인덱서, 알림 서버, 이미지 저장소, 검색 서버를 함께 쓰는 경우가 많다.

다만 핵심 자산이나 중요한 상태 변경이 블록체인과 연결된다.

```text
Web2에서는 서버가 사용자 상태의 중심이다.
Web3에서는 지갑과 체인이 사용자 상태의 중심으로 들어온다.
```

이 차이가 꽤 크다.

## 지갑은 로그인 버튼이 아니라 승인 도구에 가깝다

Web3 서비스를 처음 만들 때 가장 헷갈리는 부분이 지갑 연결이다.

버튼 이름은 보통 `Connect Wallet`이다. 그래서 마치 소셜 로그인처럼 느껴진다. 하지만 지갑 연결은 로그인과 완전히 같지 않다.

처음에는 이렇게 나누면 좋다.

| 행동 | 의미 | 체인 상태 변경 | 가스비 |
| --- | --- | --- | --- |
| 지갑 연결 | 서비스가 내 주소를 볼 수 있게 허용 | 없음 | 없음 |
| 메시지 서명 | 내가 이 주소의 소유자임을 증명 | 없음 | 없음 |
| 트랜잭션 서명 | 체인 상태 변경을 승인 | 있음 | 보통 있음 |

지갑 연결은 "이 서비스가 내 주소를 읽어도 된다"에 가깝다. 이 단계만으로 토큰이 이동하지는 않는다.

메시지 서명은 "내가 이 주소를 제어하는 사람이다"를 증명할 때 자주 쓴다. 예를 들어 로그인 요청 문구에 서명하면, 서버는 그 서명을 검증해서 이 사용자가 해당 주소를 가진 사람인지 확인할 수 있다.

트랜잭션 서명은 더 무겁다. 토큰 전송, NFT 민팅, [[swap-token-exchange|swap]], approve, 스테이킹처럼 체인 상태가 바뀌는 행동은 트랜잭션이 된다. 이때 사용자는 무엇이 실행되는지 확인하고 서명한다.

지갑 개발 관점에서는 이 차이를 UI에서 잘 보여줘야 한다.

```text
연결인가?
로그인용 메시지 서명인가?
자산이나 권한이 움직이는 트랜잭션인가?
```

사용자는 이 세 가지를 자주 헷갈린다. 그래서 지갑은 단순히 확인 버튼을 보여주는 앱이 아니라, 사용자가 무엇에 동의하는지 번역해주는 안전 장치에 가깝다.

## Web3 서비스의 기본 흐름

Web3 기반 서비스를 사용자가 실제로 쓴다고 생각해보자.

예를 들어 어떤 서비스에서 NFT를 민팅하거나, 토큰을 예치하거나, 투표에 참여한다고 해보자. 겉으로는 버튼 하나지만 내부 흐름은 보통 이렇게 움직인다.

```text
사용자가 서비스 화면을 연다.
지갑을 연결한다.
서비스가 사용자의 주소와 네트워크를 확인한다.
사용자가 어떤 행동을 선택한다.
서비스가 서명 요청 또는 트랜잭션 요청을 만든다.
지갑이 요청 내용을 보여준다.
사용자가 확인하면 지갑이 서명한다.
서명된 트랜잭션이 네트워크로 전송된다.
스마트 컨트랙트가 실행된다.
블록체인 상태가 바뀐다.
서비스 화면이 최신 상태를 다시 보여준다.
```

이 흐름에서 서비스와 지갑의 역할이 나뉜다.

서비스는 사용자가 무엇을 하려는지 알고 있다. 예를 들어 "이 NFT를 민팅한다", "이 풀에 예치한다", "이 토큰을 교환한다" 같은 의도를 만든다.

지갑은 사용자의 키를 가지고 있다. 그래서 최종 승인과 서명을 담당한다.

블록체인은 실행 결과를 기록한다. 트랜잭션이 성공하면 상태가 바뀌고, 실패하면 상태 변경은 반영되지 않는다.

처음에는 이 그림만 머릿속에 있으면 좋다.

```text
서비스 화면 -> 지갑 확인 -> 사용자 서명 -> 체인 실행 -> 화면 갱신
```

## 스마트 컨트랙트는 Web3 서비스의 실행 지점이다

Web3 서비스에서 모든 로직이 프론트엔드나 서버에만 있는 것은 아니다. 중요한 규칙은 스마트 컨트랙트에 들어갈 수 있다.

스마트 컨트랙트는 블록체인 위에 배포된 프로그램이다. 주소를 가지고 있고, 상태를 저장할 수 있고, 트랜잭션으로 함수를 실행할 수 있다.

예를 들어 이런 규칙이 컨트랙트에 들어갈 수 있다.

- 누가 NFT를 민팅할 수 있는가
- 토큰을 얼마나 예치했는가
- 투표권은 어떻게 계산되는가
- swap 가격은 어떤 풀 상태로 계산되는가
- 보상은 언제 얼마나 받을 수 있는가

서비스 화면은 컨트랙트의 상태를 읽고, 사용자가 버튼을 누르면 컨트랙트 함수를 실행하는 트랜잭션을 만든다.

그래서 Web3 서비스에서 프론트엔드는 단순한 웹 화면이 아니다. 사용자의 지갑과 컨트랙트를 연결하는 인터페이스가 된다.

```text
프론트엔드 = 사용자의 의도를 트랜잭션 요청으로 바꾸는 화면
지갑 = 그 요청을 검토하고 서명하는 도구
스마트 컨트랙트 = 체인 위에서 실제 규칙을 실행하는 코드
```

이렇게 보면 Web3 기반 서비스라는 말이 조금 덜 흐릿해진다.

## 모든 것이 온체인일 필요는 없다

Web3 기반이라고 해서 모든 데이터를 블록체인에 올려야 하는 것은 아니다.

오히려 대부분의 서비스는 온체인과 오프체인을 섞는다.

| 영역 | 온체인에 두기 좋은 것 | 오프체인에 두기 좋은 것 |
| --- | --- | --- |
| 소유권 | 토큰 잔액, NFT 소유자 | 목록 정렬, 검색 캐시 |
| 실행 규칙 | 민팅, 전송, 예치, 청구 | 추천, 알림, 운영 화면 |
| 증명 | 트랜잭션 기록, 이벤트 | 분석 데이터, 사용자 설정 |
| 비용 | 중요한 상태 변경 | 자주 바뀌는 임시 데이터 |

블록체인은 강한 신뢰와 검증 가능성을 준다. 대신 느리고 비싸고, 한 번 기록한 데이터를 쉽게 바꾸기 어렵다.

그래서 개발할 때는 "무엇을 반드시 온체인에 둬야 하는가?"를 먼저 봐야 한다.

```text
사용자 자산인가?
누가 봐도 검증 가능해야 하는 기록인가?
서비스 운영자가 마음대로 바꾸면 안 되는 규칙인가?
```

이런 것들은 온체인에 어울린다.

반대로 피드 정렬, 검색 필터, 푸시 알림 상태, 튜토리얼 완료 여부 같은 것은 꼭 체인에 올릴 필요가 없다. 서버나 로컬 DB가 더 자연스러울 수 있다.

Web3 기반 서비스는 서버가 없는 서비스라기보다, 신뢰가 필요한 부분을 온체인으로 옮긴 서비스에 가깝다.

## 자주 만나는 규격들

Web3 기반 서비스를 만들다 보면 몇 가지 이름을 자주 보게 된다.

처음부터 모두 외울 필요는 없다. 다만 "이 이름들이 어떤 문제를 다루는가" 정도는 잡아두면 좋다.

| 이름 | 간단히 보면 | 서비스에서 만나는 곳 |
| --- | --- | --- |
| ERC-20 | 대체 가능한 토큰 표준 | 토큰 잔액, 전송, approve, swap |
| ERC-721 | NFT 표준 | NFT 소유권, 전송, 민팅 |
| EIP-1193 | 지갑 provider API | 웹 앱이 지갑에 요청을 보내는 인터페이스 |
| EIP-4361 | Sign-In with Ethereum | 지갑 서명 기반 로그인 메시지 형식 |
| EIP-712 | Typed structured data signing | 사용자가 읽기 쉬운 구조화된 서명 |

예를 들어 웹 서비스가 브라우저 지갑과 연결될 때는 provider를 통해 계정 요청, 네트워크 확인, 트랜잭션 요청을 보낼 수 있다. 이때 EIP-1193 같은 인터페이스가 자주 언급된다.

로그인에서는 Sign-In with Ethereum, 줄여서 SIWE를 볼 수 있다. 비밀번호 대신 지갑 주소와 메시지 서명으로 사용자를 확인하는 방식이다.

그리고 사용자가 어떤 내용에 서명하는지 사람이 읽기 좋게 보여주기 위해 EIP-712 구조화 서명이 쓰이기도 한다.

이 규격들은 Web3 서비스를 더 예측 가능하게 만들기 위한 약속에 가깝다.

```text
서비스는 요청을 만들고,
지갑은 사용자가 이해할 수 있게 보여주고,
사용자는 자기 키로 승인한다.
```

## 지갑 개발자가 특히 봐야 하는 부분

지갑을 개발하는 입장에서는 Web3 기반 서비스의 복잡함을 사용자가 직접 다 떠안지 않게 해줘야 한다.

서비스는 요청을 보낼 수 있다. 하지만 사용자는 그 요청이 안전한지, 어떤 권한을 주는지, 어떤 자산이 움직이는지 모를 수 있다.

그래서 지갑은 아래 정보를 잘 보여줘야 한다.

- 어떤 dApp이 요청했는가
- 어떤 네트워크에서 실행되는가
- 어떤 주소가 서명하는가
- 단순 연결인지, 메시지 서명인지, 트랜잭션인지
- 토큰이 이동하는가
- approve 권한을 주는가
- 예상 가스비는 어느 정도인가
- 컨트랙트 주소를 신뢰할 수 있는가
- 서명 메시지가 사람이 읽을 수 있는 형태인가

특히 approve와 서명 메시지는 조심해서 보여줘야 한다.

approve는 사용 권한을 주는 행동이다. 사용자가 무제한 approve를 허용하면, 해당 컨트랙트가 나중에 토큰을 가져갈 수 있는 범위가 커진다.

메시지 서명은 가스비가 없고 체인 상태를 직접 바꾸지 않을 수 있다. 하지만 그렇다고 항상 안전한 것은 아니다. 어떤 메시지에 서명하는지에 따라 로그인, 주문 승인, 권한 위임 같은 의미를 가질 수 있다.

지갑은 결국 사용자의 마지막 방어선이다.

```text
서비스가 만든 요청을
사용자가 이해할 수 있는 언어로 바꿔주는 것
```

이게 Web3 지갑 UX의 핵심에 가깝다.

## Web3 기반이라는 말을 더 쉽게 바꾸면

`Web3 기반 서비스`라는 표현이 너무 넓게 느껴질 때는 더 구체적으로 바꿔보면 좋다.

예를 들어 이런 식이다.

| 추상적인 표현 | 더 구체적인 표현 |
| --- | --- |
| Web3 기반 서비스 | 지갑 연결과 온체인 트랜잭션을 사용하는 서비스 |
| Web3 로그인 | 지갑 주소와 메시지 서명으로 로그인하는 방식 |
| Web3 지갑 | 블록체인 계정의 키를 관리하고 서명하는 앱 |
| Web3 결제 | 토큰 전송으로 결제를 처리하는 흐름 |
| Web3 게임 | 아이템 소유권이나 거래 기록 일부가 온체인에 있는 게임 |
| Web3 커뮤니티 | 토큰이나 NFT 소유권으로 참여 권한을 확인하는 커뮤니티 |

이렇게 바꾸면 갑자기 덜 멀어진다.

Web3라는 말은 방향을 말한다. 하지만 개발할 때 필요한 것은 구체적인 흐름이다.

```text
누가 어떤 주소로 들어오는가?
어떤 메시지에 서명하는가?
어떤 컨트랙트를 호출하는가?
어떤 상태가 체인에 기록되는가?
사용자에게 무엇을 보여줘야 안전한가?
```

이 질문에 답할 수 있으면, Web3 기반이라는 말은 꽤 현실적인 설계 문제로 바뀐다.

## 결국 핵심은 소유와 승인이다

Web3를 "읽기, 쓰기, 소유"라고 설명하는 경우가 많다. 이 표현은 꽤 유용하다.

Web1은 주로 읽는 웹이었다. Web2는 사용자가 글을 쓰고 사진을 올리고 관계를 만들 수 있는 웹이 됐다. Web3는 여기에 소유와 직접 승인의 감각을 더한다.

하지만 개발자 관점에서는 조금 더 작게 보는 편이 좋다.

```text
사용자가 자기 주소를 가진다.
사용자가 자기 키로 승인한다.
중요한 결과가 체인에 기록된다.
다른 서비스도 그 기록을 검증할 수 있다.
```

이 네 문장이 Web3 기반 서비스를 이해하는 데 꽤 좋은 출발점이 된다.

그러면 지갑 개발을 볼 때도 방향이 잡힌다.

지갑은 코인을 담는 주머니만이 아니다. 사용자의 블록체인 계정, 자산, 권한, 서명을 다루는 인터페이스다.

그리고 Web3 기반 서비스는 그 지갑을 통해 사용자가 온체인 세계와 상호작용하게 만드는 서비스다.
