![Web3 기반 서비스란?](/images/web3-service-cover.png)

지갑을 개발하다 보면 `Web3 기반`, `Web3 서비스`, `dApp 연동` 같은 말을 자주 듣는다. 그런데 막상 "Web3 기반 서비스가 정확히 뭐예요?"라고 물으면 대답이 애매해질 때가 있다.

처음에는 "아무튼 블록체인이나 코인이 들어가면 Web3인가?"처럼 느껴진다. 나도 특히 서비스와 지갑이 잘 연결되지 않았다. 서비스는 화면과 서버처럼 보이고, 지갑은 별도 앱처럼 보이기 때문이다.

이 글은 거창한 Web3 철학을 설명하는 글이 아니다. 내가 헷갈렸던 질문들을 하나씩 붙여가면서, 개발자가 볼 수 있는 흐름으로 정리한 메모다.

```text
서비스는 무엇을 요청하고,
지갑은 무엇을 승인하고,
노드는 무엇을 검증하고,
컨트랙트는 무엇을 바꾸는가?
```

이 네 질문이 붙으면 Web3 기반 서비스라는 말이 조금 덜 멀어진다.

## 먼저 전체 그림만 보기

Web3 기반 서비스를 한 장으로 보면 이렇게 잡을 수 있다.

![Web3 서비스 전체 흐름](/images/web3-flow-overview-handdrawn.png)

한 문장으로 줄이면 이렇다.

```text
Web3 기반 서비스는 사용자의 지갑 서명을 통해 블록체인 상태를 읽고 쓰는 서비스다.
```

여기서 중요한 단어는 다섯 개다.

- 앱
- 지갑
- 서명
- 노드
- 컨트랙트

Web2 서비스에서는 보통 서버가 사용자 요청을 받아 DB를 바꾼다.

```text
사용자
-> 서비스 화면
-> 서버 API
-> DB 변경
```

Web3 서비스에서는 중요한 상태 변경 앞에 지갑과 서명이 들어온다.

```text
사용자
-> 서비스 화면
-> 지갑 확인
-> 개인키 서명
-> 노드 검증
-> 컨트랙트 실행
-> 블록체인 상태 변경
```

겉으로는 버튼 하나를 누르는 것처럼 보이지만, 내부에서는 "사용자의 의도"가 "사용자의 권한으로 승인된 트랜잭션"이 되는 과정을 지난다.

## 결국 지갑이 핵심일까

처음에는 "그럼 결국 지갑이 핵심인가?"라는 생각이 든다. 거의 맞다. 다만 더 정확히 말하면, 핵심은 지갑 그 자체가 아니라 **사용자의 소유와 승인**이다.

지갑은 그 소유와 승인을 실제로 다루는 인터페이스다.

```text
지갑 주소 = 공개 식별자
개인키 = 그 주소를 제어하는 비밀 키
지갑 앱 = 개인키로 서명해주는 도구
```

그래서 지갑은 단순히 코인을 담는 앱이 아니다. 사용자가 블록체인 위의 자산과 권한을 움직일 수 있게 해주는 승인 도구에 가깝다.

서비스가 "이 컨트랙트의 함수를 호출하자"는 요청을 만들면, 지갑은 사용자에게 그 요청을 보여준다. 사용자가 확인하면 지갑은 개인키로 서명한다. 그 서명이 있어야 네트워크는 "이 주소의 주인이 이 행동을 승인했구나"라고 판단할 수 있다.

## 여기서 소유한다는 것은 무엇일까

Web3에서 "소유한다"는 말은 지갑 앱 안에 파일처럼 무언가가 들어 있다는 뜻이 아니다. 더 정확히는 **블록체인 상태에 어떤 자산이나 권리가 내 주소에 귀속되어 있다**는 뜻이다.

예를 들어 토큰 잔액은 이런 식으로 볼 수 있다.

```text
0xAAA -> 100 TOKEN
0xBBB -> 20 TOKEN
```

NFT라면 이런 상태일 수 있다.

```text
tokenId 7 -> owner 0xAAA
```

스테이킹이나 예치도 마찬가지다.

```text
0xAAA -> 예치 수량 30
0xAAA -> 받을 수 있는 보상 5
```

즉 "내가 소유한다"는 것은 체인이나 컨트랙트 안의 상태가 내 주소를 가리키고 있다는 뜻이다. 그리고 내가 그 주소의 개인키를 가지고 있으면, 서명을 통해 그 상태를 이동하거나 사용할 수 있다.

개발자식으로는 이렇게 보면 편하다.

```text
Map<Address, Balance>
Map<TokenId, OwnerAddress>
Map<Address, Permission>
```

지갑 주소는 이 맵에서 key처럼 동작한다. 메모리 주소처럼 실제 데이터가 그 위치에 들어 있는 것은 아니지만, 체인 상태에서 내 기록을 찾는 공개 식별자에 가깝다.

## 지갑 주소는 서버가 발급하지 않는다

처음에 가장 신기했던 부분은 지갑 주소가 서버에서 발급되는 계정 번호가 아니라는 점이다.

비수탁 지갑에서는 보통 지갑 앱이 로컬에서 키를 만든다. 인터넷이 없어도 주소 생성 자체는 가능하다.

![지갑 주소 생성 흐름](/images/web3-wallet-address-generation-handdrawn.png)

흐름은 대략 이렇다.

```text
안전한 난수
-> 니모닉
-> 개인키
-> 공개키
-> 지갑 주소
```

EVM 계열 지갑에서는 보통 256비트 개인키를 사용한다. 실제 지갑에서는 사용자가 개인키를 직접 외우기 어렵기 때문에, 12개 또는 24개의 단어로 된 시드 문구를 제공하고 그 문구에서 여러 개인키를 파생한다.

```text
시드 문구
-> 마스터 키
-> 계정 1 개인키
-> 계정 2 개인키
-> 계정 3 개인키
```

그래서 주소 생성은 은행 계좌 개설처럼 중앙 서버가 번호를 부여하는 흐름과 다르다. 지갑이 충분히 좋은 난수로 개인키를 만들고, 그 개인키에서 주소를 계산한다.

여기서 중복 가능성이 궁금해질 수 있다. 이론적으로는 가능하다. 하지만 정상적인 난수 생성기를 사용하면 현실적으로 걱정할 수준이 아니다. 가능한 키 공간이 너무 크기 때문이다.

진짜 위험한 것은 "중복"보다 "약한 난수"와 "개인키 유출"이다.

```text
위험한 것:
현재 시간으로 키 만들기
사용자 생일로 키 만들기
약한 랜덤 함수 사용
테스트용 시드 문구를 실서비스에 사용
시드 문구를 외부에 노출
```

그래서 지갑 개발에서는 키 생성보다도 난수 품질, 안전한 저장, 유출 방지, 서명 화면 UX가 중요해진다.

## 서명은 개인키로 찍는 승인 도장이다

Web3에서 서명은 "내 개인키를 공개하지 않고도, 이 요청을 내가 승인했다는 것을 증명하는 값"이다.

![서명 흐름](/images/web3-signature-flow-handdrawn.png)

트랜잭션 서명 흐름은 대략 이렇다.

```text
1. 서비스가 트랜잭션 요청을 만든다.
2. 지갑이 요청 내용을 사용자에게 보여준다.
3. 사용자가 확인한다.
4. 지갑이 개인키로 요청 데이터에 서명한다.
5. 트랜잭션 데이터와 서명값이 네트워크로 전송된다.
```

중요한 점은 개인키가 밖으로 나가지 않는다는 것이다.

```text
밖으로 나가는 것:
트랜잭션 데이터 + 서명값

밖으로 나가면 안 되는 것:
개인키
시드 문구
```

노드는 서명값을 보고 이 요청이 해당 주소의 개인키로 승인된 것인지 검증할 수 있다. 개인키를 몰라도 검증은 가능하다. 공개키 암호 방식 덕분이다.

서명은 두 가지를 보장한다.

```text
인증:
이 주소의 주인이 승인한 요청인가?

무결성:
서명한 뒤 요청 내용이 바뀌지 않았는가?
```

누군가 중간에서 `to` 주소나 `amount`를 바꾸면 서명이 더 이상 맞지 않는다. 그러면 노드가 유효한 트랜잭션으로 받아들이지 않는다.

그래서 지갑의 승인 화면은 정말 중요하다. 사용자는 그 화면을 보고 "내가 무엇에 서명하는지" 이해해야 한다.

## 온체인은 Web2 서버와 같은 걸까

겉으로 보면 비슷하다. 둘 다 상태를 저장하고 요청을 받아 상태를 바꾼다.

하지만 차이는 "서버냐 아니냐"가 아니라 **누가 마음대로 바꿀 수 있느냐**에 있다.

```text
Web2 서버/DB:
서비스 운영자가 관리하는 상태

온체인:
여러 노드가 같은 규칙으로 검증하고 공유하는 상태
```

Web2 서비스에서 포인트가 있다면 서버 DB에 이런 값이 있을 수 있다.

```text
user_id 123 -> 10,000 points
```

운영자는 정책상 이 값을 수정할 수 있다. 장애가 나면 복구할 수도 있고, 관리자 기능으로 조정할 수도 있다.

반면 온체인 토큰 잔액은 이런 식으로 체인 상태에 기록된다.

```text
0xAAA -> 10,000 TOKEN
```

이 상태는 사용자의 서명이나 컨트랙트 규칙 없이는 바뀌지 않는다. 더 정확히 말하면, 누군가 자기 컴퓨터의 데이터를 조작할 수는 있어도 다른 노드들이 그 변경을 유효한 상태로 인정하지 않는다.

```text
값을 물리적으로 못 건드린다
```

라기보다,

```text
규칙에 맞지 않는 변경은 네트워크가 인정하지 않는다
```

에 가깝다.

## 노드와 네트워크는 무엇일까

노드는 블록체인 프로그램을 실행하는 컴퓨터다. 네트워크는 그런 노드들이 서로 연결된 전체 묶음이다.

```text
노드 = 장부를 들고 규칙을 검사하는 컴퓨터
네트워크 = 노드들이 서로 연결된 전체 시스템
```

노드는 트랜잭션이나 블록을 받으면 여러 가지를 확인한다.

```text
서명이 맞는가?
잔액이 충분한가?
nonce가 맞는가?
가스비 조건이 맞는가?
컨트랙트 규칙에 맞는가?
```

한 노드가 자기 데이터만 마음대로 바꾼다고 해서 네트워크의 진짜 상태가 바뀌지는 않는다. 다른 노드들이 같은 규칙으로 검증했을 때 받아들여야 한다.

그래서 블록체인의 "못 바꿈"은 마법 같은 방어막이 아니다. 여러 노드가 같은 규칙을 적용하고, 규칙에 맞는 기록만 공유하기 때문에 생기는 성질이다.

노드가 하나뿐이라면 어떨까? 그러면 Web3의 핵심 장점이 많이 사라진다.

```text
노드 1개 체인:
그 노드 운영자를 믿어야 한다.

여러 독립 노드가 검증하는 체인:
한 운영자가 마음대로 상태를 바꾸기 어렵다.
```

그래서 "블록체인을 썼는가?"보다 더 중요한 질문은 이것이다.

```text
누가 노드를 운영하는가?
검증자가 몇 명인가?
서로 독립적인가?
컨트랙트 관리자 권한은 어디까지인가?
운영진이 사용자의 자산을 임의로 움직일 수 있는가?
```

이 질문에 따라 Web3다운 정도가 달라진다.

## 블록체인, 노드, 컨트랙트, 네트워크를 연결해서 보기

이제 용어를 한 번에 연결해보자.

![노드와 컨트랙트 상태 변경 흐름](/images/web3-node-contract-state-handdrawn.png)

각 용어는 이렇게 붙는다.

```text
네트워크
= 노드들이 연결된 전체 시스템

노드
= 블록체인 프로그램을 돌리고 규칙을 검증하는 컴퓨터

블록체인
= 노드들이 함께 보관하고 검증하는 장부

컨트랙트
= 블록체인 위에 배포된 상태 변경 규칙 코드
```

컨트랙트는 서버처럼 혼자 어딘가에서 계속 실행되는 프로그램이 아니다. 컨트랙트 코드는 체인 위에 있고, 트랜잭션이 들어오면 노드들이 같은 코드를 실행해보고 결과를 검증한다.

예를 들어 NFT 민팅 버튼을 누르면 흐름은 이렇게 지나간다.

```text
1. 서비스가 mint() 호출 트랜잭션을 만든다.
2. 지갑이 사용자에게 요청 내용을 보여준다.
3. 사용자가 승인하면 지갑이 개인키로 서명한다.
4. 서명된 트랜잭션이 노드로 전송된다.
5. 노드들이 서명, 잔액, nonce, 규칙을 검증한다.
6. 컨트랙트 코드가 실행된다.
7. 규칙에 맞으면 새 블록에 포함된다.
8. 블록체인 상태가 바뀐다.
```

이 흐름을 개발자식으로 줄이면 이렇게 볼 수 있다.

```text
dApp frontend
-> wallet
-> signed transaction
-> node/RPC
-> smart contract
-> blockchain state
```

여기서 블록체인은 장부 전체이고, 블록은 그 장부의 한 페이지처럼 볼 수 있다.

```text
트랜잭션 = 장부에 새로 적어달라는 요청
블록 = 여러 트랜잭션을 묶은 장부 페이지
블록체인 = 장부 페이지들이 해시로 이어진 구조
상태 = 지금까지 장부를 계산한 현재 결과
```

과거 블록을 바꾸면 그 블록의 해시가 바뀌고, 그 뒤 블록들이 가리키는 이전 해시와 맞지 않게 된다. 그래서 과거 기록을 몰래 바꾸려면 그 이후 기록까지 다시 만들고 네트워크 다수에게 인정받아야 한다.

## Web3 서비스는 서버가 없는 서비스가 아니다

여기서 조심할 점이 있다. Web3 기반이라고 해서 서버가 전혀 없는 것은 아니다.

현실의 서비스는 보통 온체인과 오프체인을 섞는다.

| 영역 | 온체인에 두기 좋은 것 | 오프체인에 두기 좋은 것 |
| --- | --- | --- |
| 소유권 | 토큰 잔액, NFT 소유자 | 목록 정렬, 검색 캐시 |
| 실행 규칙 | 민팅, 전송, 예치, 청구 | 추천, 알림, 운영 화면 |
| 증명 | 트랜잭션 기록, 이벤트 | 분석 데이터, 사용자 설정 |
| 비용 | 중요한 상태 변경 | 자주 바뀌는 임시 데이터 |

블록체인은 검증 가능한 상태를 만드는 데 강하다. 대신 느리고 비싸고, 한 번 기록한 것을 쉽게 바꾸기 어렵다.

그래서 개발할 때는 먼저 이 질문을 해야 한다.

```text
이 상태는 운영자가 마음대로 바꾸면 안 되는가?
다른 사람도 검증할 수 있어야 하는가?
사용자 자산이나 권한과 직접 연결되는가?
```

그렇다면 온체인에 둘 이유가 생긴다.

반대로 피드 정렬, 검색, 푸시 알림, 튜토리얼 완료 여부, 화면 캐시 같은 것은 서버나 로컬 DB가 더 자연스럽다.

Web3 서비스는 서버가 없는 서비스라기보다, **신뢰가 중요한 일부 상태를 온체인으로 옮긴 서비스**에 가깝다.

## 지갑 개발자는 무엇을 봐야 할까

지갑 개발자 입장에서 핵심은 사용자의 마지막 승인 지점을 안전하게 만드는 것이다.

서비스는 요청을 만들 수 있다. 하지만 사용자는 그 요청이 어떤 의미인지 모를 수 있다.

그래서 지갑은 아래 정보를 잘 보여줘야 한다.

- 어떤 서비스가 요청했는가
- 어떤 네트워크에서 실행되는가
- 어떤 주소가 서명하는가
- 단순 연결인지, 메시지 서명인지, 트랜잭션인지
- 토큰이나 NFT가 이동하는가
- 컨트랙트에 권한을 주는가
- 예상 가스비는 어느 정도인가
- 서명 메시지가 사람이 읽을 수 있는가

특히 `approve`와 메시지 서명은 조심해야 한다.

approve는 토큰을 직접 보내는 행동이 아니라, 컨트랙트에 토큰 사용 권한을 주는 행동이다. 사용자가 무제한 approve를 허용하면 나중에 그 컨트랙트가 가져갈 수 있는 범위가 커질 수 있다.

메시지 서명은 가스비가 없고 체인 상태를 직접 바꾸지 않을 수 있다. 하지만 그렇다고 항상 안전한 것은 아니다. 로그인, 주문 승인, 권한 위임 같은 의미를 가질 수 있기 때문이다.

지갑은 결국 이 일을 한다.

```text
서비스가 만든 요청을
사용자가 이해할 수 있는 언어로 바꿔준다.
```

그래서 지갑은 단순한 확인 버튼이 아니라 사용자의 권한 경계다.

## Android 개발자 관점으로 다시 붙이기

Android 개발자가 Web3 지갑 프로젝트를 보면 처음에는 익숙한 구조와 낯선 구조가 섞여 보인다.

익숙한 부분은 여전히 있다.

```text
Screen
-> ViewModel
-> UseCase
-> Repository
```

하지만 Repository 뒤쪽에서 Web3 흐름이 이어진다.

```text
Repository
-> 트랜잭션 요청 생성
-> 개인키 로드
-> 서명
-> RPC/노드 전송
-> 컨트랙트 실행
-> 상태 조회 또는 이벤트 수신
-> 화면 갱신
```

그러니까 Web3 지갑 앱은 완전히 낯선 앱이 아니다. Android 앱 구조 위에 "키 관리, 서명, 트랜잭션, 노드, 컨트랙트, 온체인 상태"가 얹힌 것이다.

처음 코드를 읽을 때는 아래 질문으로 따라가면 좋다.

```text
1. 사용자가 어떤 버튼을 누르는가?
2. 그 버튼이 어떤 요청을 만드는가?
3. 그 요청은 메시지 서명인가, 트랜잭션인가?
4. 개인키는 어디서 로드되는가?
5. 서명된 값은 어디로 전송되는가?
6. 결과는 어떻게 앱으로 돌아오는가?
```

이 질문들이 잡히면 지갑 프로젝트가 훨씬 덜 막연해진다.

## 결국 내가 몰랐던 것

처음에는 Web3가 코인, 토큰, NFT, 블록체인 같은 단어들의 묶음처럼 보였다.

그런데 실제로 헷갈렸던 것은 용어 자체가 아니라, 각 용어가 서비스 흐름 안에서 어디에 붙는지였다.

```text
지갑은 주소와 개인키를 다룬다.
주소는 체인 상태에서 나를 찾는 공개 식별자다.
개인키는 그 주소의 권한을 행사하는 비밀 값이다.
서명은 개인키를 공개하지 않고 승인했음을 증명한다.
노드는 서명과 규칙을 검증한다.
컨트랙트는 정해진 규칙대로 상태를 바꾼다.
블록체인은 그 결과를 장부처럼 기록하고 공유한다.
```

그래서 Web3 기반 서비스라는 말을 더 쉽게 바꾸면 이렇게 말할 수 있다.

```text
사용자의 지갑 주소로 계정을 식별하고,
사용자의 개인키 서명으로 행동을 승인하며,
중요한 결과를 블록체인 상태에 기록하는 서비스
```

이 정도가 잡히면 이제 시작할 수 있다.

지갑은 코인을 담는 주머니만이 아니다. 사용자의 블록체인 계정, 자산, 권한, 서명을 다루는 인터페이스다.

그리고 Web3 서비스는 그 지갑을 통해 사용자가 온체인 세계와 상호작용하게 만드는 서비스다.
