![L1 체인, L2 체인, 전용 Layer 1 이해하기](/images/l1-l2-dedicated-layer1-handdrawn.png)

블록체인을 공부하다 보면 `L1`, `L2`, `Layer 1`, `Layer 2`, `전용 L1`, `앱체인` 같은 말을 자주 만난다. 처음에는 이름만 봐도 계층 구조처럼 보이는데, 막상 설명을 들으면 또 헷갈린다.

특히 이런 질문이 생긴다.

```text
L1이 기본 체인이라면 L2는 별도 체인인가?
L2도 체인인데 왜 L1 위에 있다고 말할까?
전용 Layer 1은 그냥 또 다른 L1인가?
```

이 글은 합의 알고리즘이나 롤업 구조를 깊게 파는 글이 아니다. [[blockchain-study-guide|블록체인 학습방법 가이드]]와 [[web3-service-basics|Web3 기반 서비스]]에서 이어지는 초심자용 메모로, "어떤 체인 위에서 거래가 처리되고, 어디에 최종 기록되는가"를 중심으로 정리해보려 한다.

## 먼저 한 문장으로 보기

세 표현을 한 문장씩 줄이면 이렇게 볼 수 있다.

```text
L1 체인:
자기 네트워크, 자기 검증자, 자기 장부를 가진 기본 블록체인

L2 체인:
L1의 보안과 최종 기록을 활용하면서 거래 처리를 더 빠르고 싸게 하려는 확장 체인

전용 Layer 1:
특정 앱, 게임, 서비스, 생태계를 위해 따로 만든 독립 L1 체인
```

처음에는 아래 구분만 잡아도 충분하다.

| 구분 | 짧은 설명 | 예시 |
| --- | --- | --- |
| L1 | 독립적인 기본 체인 | Bitcoin, Ethereum, Solana, Avalanche |
| L2 | L1 위에 붙은 확장 체인 | Arbitrum, Optimism, Base, zkSync |
| 전용 L1 | 특정 목적에 맞춘 독립 체인 | 게임 전용 체인, 서비스 전용 체인, 앱체인 |

핵심은 "체인이 몇 층인가"보다 **누가 보안을 책임지고, 어디에 최종 상태가 기록되는가**다.

## L1은 기본 장부를 가진 독립 체인이다

L1은 가장 기본이 되는 블록체인이다. 자체 네트워크가 있고, 자체 규칙이 있고, 트랜잭션을 검증하는 노드나 검증자 집합이 있다.

예를 들어 Ethereum을 보면 이런 역할을 모두 직접 한다.

```text
트랜잭션 받기
-> 서명과 잔액 검증
-> 스마트 컨트랙트 실행
-> 블록 생성
-> 상태 변경 기록
```

[[simple-payment-blockchain-signature|블록체인 서명]] 관점에서 보면, 사용자가 서명한 트랜잭션을 L1 네트워크가 직접 검증하고 처리한다. 토큰 전송, NFT 민팅, 컨트랙트 호출 같은 상태 변경도 L1의 장부에 남는다.

그래서 L1은 비유하면 "도시의 본도로이자 공식 등기부"에 가깝다.

```text
L1 = 거래를 처리하는 도로 + 결과를 기록하는 장부 + 규칙을 검증하는 네트워크
```

장점은 명확하다. 독립성이 높고, 그 체인의 보안 모델을 직접 가진다. 하지만 사용자가 많아지고 트랜잭션이 몰리면 수수료가 비싸지거나 처리 속도가 느려질 수 있다.

이 지점에서 L2가 나온다.

## L2는 L1을 확장하기 위한 보조 처리 공간이다

L2는 L1 위에 붙어서 거래를 더 빠르고 저렴하게 처리하려는 체인이다. "위에 붙는다"는 표현 때문에 물리적으로 얹혀 있는 것처럼 느껴질 수 있지만, 실제로는 보안과 최종 기록의 일부를 L1에 의존한다는 뜻에 가깝다.

아주 단순화하면 흐름은 이렇다.

```text
사용자 거래
-> L2에서 빠르게 처리
-> 여러 거래 결과를 묶음
-> 요약 결과나 증명 데이터를 L1에 기록
```

L2의 대표적인 방식 중 하나가 롤업이다. 롤업은 많은 거래를 L2에서 처리한 뒤, 그 결과를 압축해서 L1에 올리는 방식으로 이해하면 된다.

```text
L1에서 모든 거래를 하나씩 처리:
tx1, tx2, tx3, tx4, tx5 ...

L2에서 먼저 처리한 뒤 L1에 정리해서 기록:
[tx1~tx1000 처리 결과] -> L1에 제출
```

그래서 L2는 "L1을 버리고 새 체인을 쓰자"가 아니라 "L1의 장점을 유지하면서 실행 공간을 넓히자"에 가깝다.

사용자 입장에서는 L2에서 거래하면 보통 수수료가 낮고 속도가 빠르게 느껴진다. 하지만 자산을 L1에서 L2로 옮기는 브릿지, 출금 대기 시간, 사용하는 L2의 보안 방식 같은 추가 개념도 생긴다.

## L2도 체인인데 왜 L1에 의존한다고 할까

여기서 가장 많이 헷갈리는 부분이 있다.

```text
L2도 블록이 있고 트랜잭션이 있는데,
그럼 그냥 별도 L1 아닌가?
```

겉으로는 별도 체인처럼 보일 수 있다. 지갑에서 네트워크를 바꾸고, L2용 블록 탐색기도 있고, 그 안에서 토큰 전송과 컨트랙트 호출도 한다.

하지만 L2라고 부르는 이유는 최종 보안 모델이 L1과 연결되어 있기 때문이다.

단순하게 말하면 L2는 이런 약속을 가진다.

```text
거래 실행은 L2에서 빠르게 한다.
하지만 최종 정산과 분쟁 해결의 기준은 L1에 둔다.
```

물론 모든 L2가 완전히 같은 구조는 아니다. Optimistic Rollup, ZK Rollup처럼 증명 방식도 다르고, 운영 성숙도도 다르다. 그래서 실무에서는 "L2니까 무조건 안전하다"가 아니라, 그 L2가 어떤 데이터를 L1에 올리는지, 출금은 어떻게 되는지, 업그레이드 권한은 누구에게 있는지 봐야 한다.

초심자 단계에서는 이 정도로 잡으면 된다.

```text
L2는 독립 서비스처럼 보이지만,
L1을 최종 기준점으로 삼는 확장 네트워크다.
```

## 전용 Layer 1은 특정 목적을 위해 만든 독립 체인이다

전용 Layer 1은 말 그대로 특정 프로젝트나 서비스에 맞춰 만든 독립 L1이다. 게임, 거래소, 콘텐츠 서비스, 특정 Web3 생태계가 자기 목적에 맞는 체인을 만들 수 있다.

예를 들어 게임 서비스를 생각해보자.

게임 안에서는 아이템 이동, 캐릭터 성장, NFT 발행, 마켓 거래처럼 많은 상태 변경이 일어날 수 있다. 이 모든 거래를 범용 L1에 올리면 수수료와 속도 문제가 생길 수 있다.

그래서 프로젝트가 이런 선택을 할 수 있다.

```text
우리 게임의 거래는 우리 전용 체인에서 처리하자.
수수료 정책도 우리가 정하고,
블록 생성 속도도 게임 UX에 맞추고,
필요한 기능도 체인 수준에서 붙이자.
```

이런 전용 L1은 특정 서비스에 맞게 최적화하기 좋다. 다만 L2처럼 큰 L1의 보안을 그대로 빌리는 구조가 아닐 수 있다. 자체 검증자, 자체 네트워크, 자체 생태계를 만들어야 한다.

장점과 부담이 같이 온다.

| 항목 | 전용 L1의 특징 |
| --- | --- |
| 장점 | 수수료, 속도, 규칙, 기능을 서비스에 맞게 설계할 수 있다 |
| 부담 | 자체 보안, 검증자, 인프라, 브릿지, 생태계를 구축해야 한다 |

즉 전용 L1은 "우리 서비스에 딱 맞는 왕국을 만들자"에 가깝다. 대신 그 왕국의 도로, 법, 경비, 출입문을 직접 관리해야 한다.

## L1, L2, 전용 L1을 선택하는 기준

서비스 개발 관점에서는 "무엇이 더 최신인가"보다 "무엇을 포기하고 무엇을 얻는가"가 중요하다.

L1을 직접 쓰면 가장 단순하다. 사용자는 잘 알려진 메인 체인에서 거래하고, 자산도 그 체인 위에 있다. 대신 수수료와 처리량 문제가 UX를 흔들 수 있다.

L2를 쓰면 수수료와 속도 면에서 좋아질 수 있다. Ethereum 생태계와 가까운 상태로 확장성을 얻고 싶을 때 자연스럽다. 대신 브릿지, 출금, L2별 보안 모델을 이해해야 한다.

전용 L1을 만들면 서비스에 맞춘 자유도가 커진다. 게임이나 특정 대규모 서비스처럼 트랜잭션 패턴이 뚜렷하고 UX 요구가 강하면 매력적일 수 있다. 대신 체인 자체를 운영하는 부담이 생긴다.

표로 보면 이렇게 정리할 수 있다.

| 선택지 | 얻는 것 | 조심할 것 |
| --- | --- | --- |
| L1 사용 | 높은 인지도, 단순한 구조, 강한 네트워크 효과 | 수수료, 처리량, UX 비용 |
| L2 사용 | 낮은 수수료, 빠른 처리, L1과 연결된 보안 | 브릿지, 출금, L2별 구조 차이 |
| 전용 L1 구축 | 서비스 맞춤 설계, 독립 정책, 높은 자유도 | 자체 보안, 운영, 생태계 확보 |

그래서 "L2가 무조건 좋다"거나 "전용 L1이 더 고급이다"처럼 보면 어렵다. 목적이 다르면 좋은 선택도 달라진다.

## 사용자 입장에서는 무엇이 달라질까

사용자에게 가장 크게 보이는 차이는 네트워크 선택, 수수료, 자산 이동이다.

같은 지갑을 쓰더라도 Ethereum, Arbitrum, Base, 어떤 전용 체인처럼 네트워크를 바꿔야 할 수 있다. 사용자가 보는 주소는 비슷해도, 자산은 어느 네트워크에 있는지에 따라 다르게 보인다.

예를 들어 같은 USDC라고 해도 이런 식으로 나뉠 수 있다.

```text
Ethereum 위의 USDC
Arbitrum 위의 USDC
Base 위의 USDC
어떤 전용 L1 위의 USDC
```

이름이 같다고 항상 같은 위치에 있는 것은 아니다. 그래서 지갑이나 서비스 화면에서는 "어떤 체인에 있는 자산인가"를 정확히 보여줘야 한다.

특히 브릿지를 사용할 때는 더 조심해야 한다.

```text
L1 자산을 L2로 옮기는가?
L2 자산을 L1로 빼는가?
전용 L1과 다른 체인을 연결하는가?
```

이 질문을 놓치면 "토큰은 있는데 다른 네트워크에 있어서 쓸 수 없는" 상황이 생긴다.

## 개발자 입장에서는 무엇을 봐야 할까

개발자가 L1, L2, 전용 L1을 볼 때는 먼저 네 가지를 확인하면 좋다.

```text
1. 트랜잭션은 어디에서 실행되는가?
2. 최종 상태는 어디에 기록되는가?
3. 보안은 어떤 네트워크나 검증자 집합에 의존하는가?
4. 사용자는 자산을 어떻게 입출금하거나 이동하는가?
```

이 네 질문이 잡히면 문서에서 낯선 표현이 나와도 덜 흔들린다.

예를 들어 어떤 서비스가 "우리는 Ethereum L2 위에서 동작합니다"라고 말하면, Ethereum을 최종 기준점으로 삼으면서 L2에서 실행 비용을 낮추려는 구조라고 볼 수 있다.

반대로 "우리 서비스는 자체 Layer 1을 사용합니다"라고 말하면, 이 서비스가 자기 목적에 맞춘 독립 체인을 운영하거나 그 생태계 위에 있다는 뜻일 수 있다. 이때는 그 체인의 검증자, 브릿지, 지갑 지원, 블록 탐색기, 수수료 토큰까지 같이 봐야 한다.

## 한 번 더 짧게 정리하기

마지막으로 가장 짧게 정리하면 이렇다.

```text
L1:
기본 블록체인. 자체 보안과 장부를 가진다.

L2:
L1을 확장하는 실행 공간. 빠르고 싸게 처리하고 L1에 정산한다.

전용 L1:
특정 서비스나 생태계를 위해 만든 독립 체인. 자유도는 크지만 운영 책임도 크다.
```

처음 공부할 때는 "몇 층인가"보다 이 질문을 먼저 던지는 편이 좋다.

```text
이 거래는 어디서 실행되고,
그 결과는 어디에 최종 기록되며,
누가 그 기록을 검증하는가?
```

이 질문에 답할 수 있으면 L1, L2, 전용 Layer 1이라는 말이 단순한 유행어가 아니라 블록체인 서비스 구조를 읽는 기준으로 바뀐다.
