![간편결제와 블록체인 서명 이해하기](/images/simple-payment-blockchain-signature-cover.png)

블록체인 지갑을 처음 쓰다 보면 `서명`, `승인`, `트랜잭션` 같은 단어가 자주 나온다. 버튼을 누르면 지갑 앱이 열리고, 사용자는 내용을 확인한 다음 승인한다.

이 흐름은 평소에 쓰는 간편결제와 꽤 비슷하게 느껴진다.

간편결제에서도 결제 버튼을 누르면 앱이 열리고, 금액을 확인하고, 지문이나 비밀번호로 승인한다. 그래서 처음에는 이런 질문이 자연스럽게 생긴다.

```text
간편결제도 내가 승인하고,
블록체인도 내가 승인하는 것 같은데 무엇이 다른 걸까?
```

이 글은 결제망이나 암호학을 깊게 설명하는 글은 아니다. [[web3-service-basics|Web3 기반 서비스]]를 볼 때 헷갈리기 쉬운 "사용자가 승인한다"는 말을 간편결제와 블록체인 서명으로 나눠보는 메모에 가깝다.

## 먼저 한 문장으로 보기

간편결제와 블록체인 서명을 한 문장씩 줄이면 이렇게 볼 수 있다.

```text
간편결제는 사용자가 본인 인증을 하면 결제사와 금융기관이 결제를 처리하는 흐름이다.

블록체인 서명은 사용자의 개인키 서명을 네트워크가 검증하고 상태를 바꾸는 흐름이다.
```

둘 다 화면에서는 "확인"처럼 보인다. 하지만 그 확인이 시스템 안에서 가지는 의미가 다르다.

처음에는 아래 구분만 잡아도 좋다.

```text
간편결제: 인증한 뒤 결제 처리를 맡긴다.
블록체인 서명: 서명한 거래를 네트워크가 검증한다.
```

## 왜 헷갈릴까

사용자 입장에서는 두 흐름이 비슷하게 보인다.

간편결제는 대략 이런 순서로 지나간다.

```text
결제 버튼
-> 결제 앱 열림
-> 금액 확인
-> 지문 또는 비밀번호 인증
-> 결제 완료
```

블록체인 서비스도 겉으로는 비슷하다.

```text
dApp 버튼
-> 지갑 앱 열림
-> 요청 내용 확인
-> 서명 또는 승인
-> 트랜잭션 전송
```

그래서 "둘 다 내가 직접 승인하는 것 아닌가?"라고 느끼기 쉽다.

맞다. 둘 다 사용자가 확인한다. 다만 그 다음에 누가 어떤 기준으로 처리하는지가 다르다.

## 간편결제는 인증 후 처리 요청에 가깝다

간편결제를 아주 단순하게 보면 이런 구조다.

```text
사용자
-> 간편결제 앱에서 본인 인증
-> 결제사 서버
-> 카드사 또는 은행
-> 결제 결과 전달
```

예를 들어 10,000원을 결제한다고 해보자. 사용자는 앱에서 금액을 보고 지문이나 비밀번호로 인증한다.

이 인증은 "내가 이 결제를 하겠다고 확인했다"는 신호다. 그 다음에는 결제사와 카드사, 은행 같은 시스템이 실제 결제 처리를 맡는다.

여기서 중요한 점은 사용자의 지문이 직접 은행 장부를 바꾸는 것은 아니라는 점이다.

```text
내 인증 = 결제를 진행해도 된다는 확인
실제 처리 = 결제사, 카드사, 은행 시스템
```

그래서 간편결제는 사용자에게 편한 입구를 제공한다. 카드번호를 매번 입력하지 않아도 되고, 복잡한 결제망 연결은 서비스가 대신 처리한다.

## 블록체인 서명은 거래의 권한 증거에 가깝다

블록체인에서는 흐름이 조금 다르다.

내 지갑 주소에 10코인이 있고, 친구 주소로 3코인을 보낸다고 해보자. 지갑 앱은 이런 거래 내용을 만든다.

```text
A 주소에서 B 주소로 3코인을 보낸다.
```

그런데 이런 문장은 아무나 만들 수 있다. 누군가 마음대로 "A 주소에서 내 주소로 10코인을 보낸다"는 거래를 만들어낼 수도 있다.

그래서 블록체인은 확인해야 한다.

```text
이 거래가 정말 A 주소의 주인이 승인한 거래인가?
```

이때 필요한 것이 개인키 서명이다. [[eoa-externally-owned-account|EOA]]에서는 개인키를 가진 사람이 트랜잭션에 서명할 수 있다. 지갑은 사용자가 내용을 확인하면 개인키로 서명하고, 서명된 트랜잭션을 네트워크로 보낸다.

흐름은 이렇게 볼 수 있다.

```text
사용자
-> 지갑에서 트랜잭션 확인
-> 개인키로 서명
-> 서명된 트랜잭션 전송
-> 네트워크가 서명 검증
-> 블록체인 상태 변경
```

네트워크는 내 개인키를 가져가는 것이 아니다. 서명 결과를 보고 "이 서명이 해당 주소를 제어하는 개인키에서 나온 것이 맞는가"를 확인한다.

검증이 맞고 잔액, nonce, gas 같은 조건도 맞으면 트랜잭션이 블록에 포함될 수 있다. 그러면 상태가 바뀐다.

```text
A 주소: 10코인 -> 7코인
B 주소: 0코인 -> 3코인
```

즉 블록체인 서명은 단순한 앱 잠금 해제가 아니다.

```text
이 계정의 주인이 이 상태 변경을 승인했다는 암호학적 증거
```

에 가깝다.

## 승인과 처리 주체를 나눠서 보기

처음에는 "승인"이라는 단어를 둘로 나눠보면 덜 헷갈린다.

| 구분 | 사용자가 하는 일 | 실제 처리 주체 | 결과가 기록되는 곳 |
| --- | --- | --- | --- |
| 간편결제 | 지문, 비밀번호, 앱 인증 | 결제사, 카드사, 은행 | 결제사와 금융기관 시스템 |
| 블록체인 서명 | 개인키로 서명 | 블록체인 네트워크, 검증자, 노드 | 블록체인 상태 |

간편결제에서는 사용자가 직접 인증하지만, 실제 결제 승인과 정산은 결제망 안에서 처리된다.

블록체인에서는 사용자가 직접 서명하고, 그 서명이 트랜잭션의 권한 증거가 된다. 네트워크는 그 증거가 맞는지 검증한다.

그래서 같은 확인 버튼처럼 보여도 중심이 다르다.

```text
간편결제의 확인 = 결제사에 보내는 인증 신호
블록체인 서명 = 네트워크가 검증할 수 있는 권한 증거
```

## 지갑 앱은 결제사가 아니다

간편결제 앱은 결제 흐름을 운영하는 서비스에 가깝다. 사용자는 카드나 계좌를 등록하고, 앱은 결제사와 금융기관의 시스템에 연결된다.

반면 [[non-custodial-wallet|논커스토디얼 지갑]]은 보통 결제사가 아니다. 지갑은 사용자의 개인키를 관리하고, 요청 내용을 보여주고, 사용자가 승인하면 서명하는 도구에 가깝다.

```text
간편결제 앱 = 결제 요청을 처리 시스템으로 연결하는 서비스
블록체인 지갑 = 개인키로 서명하는 도구
```

지갑 앱 안에 코인이 파일처럼 들어 있는 것도 아니다. 실제 기록은 블록체인 상태에 있고, 지갑은 그 상태를 바꾸기 위한 서명을 만든다.

이 관점이 잡히면 지갑이 하는 일을 조금 더 정확히 볼 수 있다.

```text
지갑은 요청을 보여준다.
사용자는 내용을 확인한다.
지갑은 개인키로 서명한다.
네트워크는 서명을 검증하고 상태를 바꾼다.
```

## 메시지 서명과 트랜잭션 서명

블록체인 지갑에서 뜨는 서명이 항상 코인을 보내는 것은 아니다.

처음에는 두 가지를 나눠서 보면 좋다.

| 구분 | 의미 | 체인 상태 변경 | 가스비 |
| --- | --- | --- | --- |
| 메시지 서명 | 내가 이 주소를 제어한다는 증명 또는 특정 문구 승인 | 보통 없음 | 보통 없음 |
| 트랜잭션 서명 | 코인 전송, 컨트랙트 실행, approve 같은 상태 변경 요청 | 있음 | 보통 있음 |

메시지 서명은 로그인에 가까운 흐름에서 쓰일 수 있다. 예를 들어 서비스가 "이 주소의 주인인지 확인하려고 합니다"라는 문구에 서명을 요청할 수 있다.

트랜잭션 서명은 더 무겁다. 토큰 전송, [[swap-token-exchange|swap]], NFT 민팅, `approve`처럼 체인 상태나 권한이 바뀌는 행동은 트랜잭션으로 이어질 수 있다.

그래서 지갑 화면이 뜨면 먼저 이렇게 물어봐야 한다.

```text
이건 단순 메시지인가?
아니면 체인 상태를 바꾸는 트랜잭션인가?
```

## 간편결제처럼 쉬워질 수는 있다

Web3 서비스도 사용자는 간편결제처럼 편하게 쓰고 싶어 한다. 매번 긴 주소, gas, nonce, approve, 컨트랙트 호출을 읽고 싶지는 않다.

그래서 지갑과 스마트 계정 쪽에서는 사용성 장벽을 낮추려는 시도가 많다.

- 지갑이 트랜잭션 내용을 사람이 읽기 쉽게 보여준다.
- EIP-712 같은 구조화 서명으로 메시지를 정리한다.
- 스마트 컨트랙트 지갑이 복구, 한도, 여러 명 승인 같은 정책을 제공한다.
- 계정 추상화 흐름이 수수료 대납이나 묶음 실행 같은 UX를 만든다.

다만 UX가 쉬워져도 기본 구조는 다르다.

```text
간편결제는 결제사가 처리한다.
블록체인은 사용자의 서명을 네트워크가 검증한다.
```

이 차이는 계속 남아 있다.

## 지갑 서명 화면에서 볼 것

간편결제에서는 보통 금액과 가맹점 이름을 먼저 확인한다. 블록체인 지갑에서는 여기에 몇 가지가 더 붙는다.

- 어떤 사이트나 앱이 요청했는가
- 어떤 지갑 주소가 서명하는가
- 어떤 네트워크에서 실행되는가
- 메시지 서명인가, 트랜잭션 서명인가
- 코인이나 토큰이 이동하는가
- `approve`처럼 토큰 사용 권한을 주는가
- 받는 주소나 컨트랙트 주소가 낯설지 않은가
- 예상 gas가 너무 이상하지 않은가

처음부터 모든 필드를 완벽히 읽을 필요는 없다. 하지만 "무엇에 서명하는가"를 보는 습관은 꼭 필요하다.

초심자에게 가장 위험한 오해는 이것이다.

```text
서명은 전부 로그인 같은 것이다.
```

어떤 서명은 로그인에 가깝다. 하지만 어떤 서명은 자산 이동이나 권한 부여로 이어질 수 있다.

## 처음에는 이렇게 외우기

처음 공부할 때는 아래 문장만 잡아도 좋다.

```text
간편결제는 내가 인증하고 결제사가 처리한다.

블록체인은 내가 서명하고 네트워크가 검증해 장부에 반영한다.
```

조금 더 풀면 이렇게 볼 수 있다.

| 질문 | 간편결제 | 블록체인 서명 |
| --- | --- | --- |
| 내가 하는 일 | 본인 인증 | 개인키 서명 |
| 앱의 역할 | 결제 요청 연결 | 요청 확인과 서명 |
| 처리 기준 | 결제사와 금융기관 시스템 | 네트워크의 서명 검증과 상태 규칙 |
| 조심할 것 | 금액, 가맹점, 자동결제 | 주소, 네트워크, 권한, 트랜잭션 내용 |

결국 둘은 사용자 경험은 닮았지만 신뢰하는 구조가 다르다.

간편결제는 결제사와 금융기관을 믿고 편하게 쓰는 흐름에 가깝다. 블록체인 서명은 사용자의 키와 네트워크 검증을 중심으로 상태를 바꾸는 흐름에 가깝다.

이 차이를 이해하면 지갑에서 뜨는 서명 화면도 조금 덜 막연하게 보인다.
