![Uniswap 이해하기](/images/uniswap-dex-cover.png)

DeFi 자료를 읽다 보면 `swap`, `pool`, `route`, `hop` 같은 단어가 같이 나온다. 처음에는 단어 하나만 궁금해서 찾아봤는데, 막상 보면 Uniswap이라는 이름이 계속 따라붙는다.

Uniswap은 블록체인에서 토큰을 교환할 때 가장 자주 언급되는 프로토콜 중 하나다. 겉으로는 거래소처럼 보인다. 토큰 A를 넣고 토큰 B를 받는 화면이 있고, 사용자는 스왑 버튼을 누른다.

그런데 내부 구조는 중앙화 거래소와 다르다.

이 글은 Uniswap을 깊은 DeFi 전략이나 수학으로 설명하는 글은 아니다. [[blockchain-study-guide|블록체인 학습방법 가이드]]에서 봤던 지갑, 트랜잭션, 스마트 컨트랙트 흐름 위에 Uniswap을 얹어보는 초심자용 메모에 가깝다.

## 먼저 한 문장으로 보기

Uniswap을 한 문장으로 줄이면 이렇게 볼 수 있다.

```text
Uniswap은 스마트 컨트랙트와 유동성 풀을 이용해 토큰을 교환하는 탈중앙화 거래소 프로토콜이다.
```

여기서 중요한 단어는 네 개다.

- 스마트 컨트랙트
- 유동성 풀
- 토큰 교환
- 탈중앙화 거래소

Uniswap을 처음 볼 때는 "코인을 사고파는 앱" 정도로만 생각하기 쉽다. 하지만 조금 더 정확히 보면, Uniswap은 사용자의 지갑과 온체인 스마트 컨트랙트가 상호작용해서 토큰 교환을 실행하는 구조다.

사용자는 거래소 계정에 로그인해서 내부 장부를 바꾸는 것이 아니다. [[non-custodial-wallet|논커스토디얼 지갑]]으로 트랜잭션에 서명하고, 그 결과가 블록체인 상태에 반영된다.

## DEX는 중앙화 거래소와 무엇이 다를까

Uniswap은 DEX라고 부른다. DEX는 `Decentralized Exchange`의 줄임말이고, 보통 탈중앙화 거래소라고 번역한다.

처음에는 중앙화 거래소와 비교해서 보는 편이 쉽다.

| 구분 | 중앙화 거래소 | Uniswap 같은 DEX |
| --- | --- | --- |
| 계정 | 거래소 계정 | 지갑 주소 |
| 자산 보관 | 거래소가 보관하는 경우가 많음 | 사용자가 지갑으로 직접 제어 |
| 거래 기록 | 거래소 내부 장부 중심 | 블록체인 트랜잭션 중심 |
| 가격 형성 | 주문서, 마켓메이커 | 유동성 풀, AMM |
| 출금 | 거래소 출금 절차 필요 | 스왑 결과가 지갑에 직접 반영 |
| 비용 | 거래소 수수료, 출금 수수료 | 스왑 수수료, 네트워크 gas |

중앙화 거래소에서는 보통 사용자가 거래소에 자산을 입금한다. 그 다음 거래소 화면에서 매수, 매도를 하면 내부 시스템이 잔액을 바꿔준다. 체인 위 전송은 입금이나 출금 시점에만 보일 수 있다.

Uniswap에서는 흐름이 다르다. 사용자가 지갑을 연결하고, 스왑 트랜잭션에 서명한다. 트랜잭션이 성공하면 내 지갑 주소의 토큰 잔액과 풀의 토큰 잔액이 바뀐다.

그래서 Uniswap을 볼 때는 거래소 화면보다 먼저 이 질문을 잡는 것이 좋다.

```text
이 행동은 어떤 스마트 컨트랙트의 상태를 바꾸는가?
```

## 앱, 프로토콜, Labs를 나눠서 보기

Uniswap이라는 이름은 여러 문맥에서 쓰인다. 이걸 섞어 생각하면 조금 헷갈린다.

처음에는 이렇게 나눠보면 좋다.

| 이름 | 의미 |
| --- | --- |
| Uniswap Protocol | 온체인에서 동작하는 스마트 컨트랙트 프로토콜 |
| Uniswap App | 사용자가 스왑하기 쉽게 만든 웹 인터페이스 |
| Uniswap Labs | Uniswap 관련 제품과 인터페이스를 만드는 회사 |
| UNI | Uniswap 거버넌스 토큰 |

사용자가 보통 접하는 것은 Uniswap App이다. 하지만 실제 스왑은 앱 화면 안에서만 끝나는 일이 아니다. 앱은 사용자가 트랜잭션을 만들고 지갑으로 서명할 수 있게 도와주는 인터페이스에 가깝다.

실제 상태 변경은 프로토콜의 스마트 컨트랙트에서 일어난다.

이 구분은 블록체인 서비스를 볼 때 꽤 중요하다. 프론트엔드 화면, 회사, 스마트 컨트랙트, 거버넌스 토큰이 항상 같은 의미는 아니기 때문이다.

## Swap은 매수와 매도보다 교환에 가깝다

Uniswap에서 가장 자주 하는 행동은 swap이다.

예를 들어 사용자가 `ETH`를 넣고 `USDC`를 받는다고 해보자. 중앙화 거래소에 익숙하면 "ETH를 팔고 USDC를 산다"라고 표현할 수 있다. 하지만 Uniswap UI에서는 보통 swap, 즉 교환으로 표현한다.

흐름은 대략 이렇다.

```text
사용자 지갑
-> 스왑할 토큰과 수량 선택
-> 예상 수령량 확인
-> 지갑에서 트랜잭션 서명
-> Uniswap 컨트랙트 실행
-> 풀의 토큰 잔액 변경
-> 사용자 지갑 잔액 변경
```

여기서 사용자의 시작점은 [[eoa-externally-owned-account|EOA]]일 수 있다. 지갑은 트랜잭션 내용을 보여주고, 사용자는 개인키로 서명한다.

즉 Uniswap의 swap은 단순한 화면 이벤트가 아니다.

```text
내 지갑이 스마트 컨트랙트에 토큰 교환을 요청하는 트랜잭션
```

으로 보는 편이 더 정확하다.

## 유동성 풀은 토큰이 모여 있는 장소다

Uniswap의 핵심은 유동성 풀이다.

중앙화 거래소에서는 보통 주문서가 있다. 누군가는 특정 가격에 팔겠다고 주문을 올리고, 누군가는 특정 가격에 사겠다고 주문을 올린다. 거래소는 이 주문들을 매칭한다.

Uniswap은 기본적으로 다른 방식으로 동작한다. 토큰을 교환하려는 사람이 매번 반대편 거래자를 직접 찾는 것이 아니라, 미리 토큰이 모여 있는 풀과 거래한다.

예를 들어 `ETH / USDC` 풀이 있다고 해보자.

```text
ETH / USDC 풀

ETH가 들어 있다.
USDC도 들어 있다.
사용자는 ETH를 넣고 USDC를 가져가거나,
USDC를 넣고 ETH를 가져갈 수 있다.
```

이 풀에 토큰을 넣는 사람을 유동성 공급자, 또는 LP라고 부른다. LP는 풀에 자산을 예치하고, 스왑이 일어날 때 발생하는 수수료를 일정한 방식으로 나눠 받을 수 있다.

버전별로 유동성을 넣는 방식과 가격 범위 설정은 다를 수 있다. 하지만 처음에는 아래 감각만 잡아도 충분하다.

```text
Uniswap의 거래 상대는 특정 개인 주문이 아니라 유동성 풀이다.
```

## AMM은 가격을 자동으로 움직이는 규칙이다

Uniswap을 설명할 때 AMM이라는 단어도 자주 나온다. AMM은 `Automated Market Maker`의 줄임말이다.

조금 단순하게 말하면 AMM은 유동성 풀 안의 토큰 비율을 보고 가격을 자동으로 조정하는 규칙이다.

예를 들어 `ETH / USDC` 풀에서 사람들이 계속 ETH를 가져가고 USDC를 넣는다고 해보자. 그러면 풀 안의 ETH는 줄고 USDC는 늘어난다. ETH가 풀 안에서 상대적으로 부족해졌으니, 다음 사람이 ETH를 가져가려면 더 비싼 가격을 치르게 된다.

반대로 사람들이 ETH를 많이 넣고 USDC를 가져가면 풀 안의 ETH는 많아지고 USDC는 줄어든다. 그러면 ETH의 풀 내 가격은 반대 방향으로 움직인다.

처음에는 수식보다 이 감각이 더 중요하다.

```text
풀에서 많이 빠져나가는 토큰은 비싸지고,
풀에 많이 들어오는 토큰은 싸지는 방향으로 가격이 움직인다.
```

Uniswap v2를 설명할 때는 `x * y = k`라는 constant product 공식을 자주 본다. 여기서 `x`와 `y`는 풀에 들어 있는 두 토큰의 양이고, `k`는 그 곱이다.

처음부터 이 공식을 깊게 계산할 필요는 없다. 다만 Uniswap에서는 가격표가 따로 고정되어 있는 것이 아니라, 풀의 상태와 규칙에 따라 교환 비율이 계산된다는 점을 이해하면 된다.

## 가격 영향과 슬리피지

Uniswap에서 스왑할 때는 price impact와 slippage도 같이 봐야 한다.

price impact는 내 거래가 풀의 가격을 얼마나 움직이는지에 가깝다. 풀에 유동성이 아주 많고 내 거래가 작으면 가격 영향은 작다. 반대로 풀은 작은데 내가 큰 거래를 하면 풀의 비율이 크게 바뀌고, 예상보다 불리한 가격을 받을 수 있다.

slippage는 사용자가 예상한 가격과 실제 체결 가격 사이의 차이다. 블록체인에서는 내가 트랜잭션을 보낸 순간과 실제로 블록에 포함되는 순간 사이에 시간이 있다. 그 사이 다른 거래가 먼저 들어가면 풀의 상태가 바뀔 수 있다.

그래서 Uniswap 같은 DEX에서는 보통 최소 수령량을 설정한다.

```text
예상 수령량: 1,000 USDC
최소 수령량: 995 USDC
```

이런 식으로 설정해두면, 실제 결과가 최소 수령량보다 나빠질 때 트랜잭션이 실패하도록 만들 수 있다. 실패해도 gas가 일부 소모될 수 있다는 점은 따로 조심해야 한다.

## Hop은 스왑 경로의 한 단계다

처음 궁금했던 `hop`은 여기서 나온다.

Uniswap에서 토큰 A를 토큰 B로 바꿀 때 항상 직접 풀만 쓰는 것은 아니다. 직접 풀의 유동성이 부족하거나, 중간 토큰을 거치는 편이 더 나은 가격을 줄 수도 있다.

예를 들어 이런 경로가 있다.

```text
USDC -> ETH
```

이 경로는 `USDC / ETH` 풀 하나를 지나므로 1 hop으로 볼 수 있다.

반면 이런 경로도 가능하다.

```text
USDC -> WETH -> UNI
```

이 경우에는 `USDC / WETH` 풀을 한 번 지나고, `WETH / UNI` 풀을 한 번 더 지난다. 그래서 2 hops다.

간단히 말하면 이렇게 볼 수 있다.

```text
hop = 스왑 경로에서 유동성 풀 하나를 지나는 단계
```

중간 토큰을 거치는 경로를 multi-hop swap이라고 부른다.

처음에는 굳이 왜 돌아가는지 이상하게 느껴질 수 있다. 하지만 DEX에서는 직접 길이 항상 가장 좋은 길이 아니다. 직접 풀은 작지만 `WETH`, `USDC`, `USDT` 같은 주요 토큰을 거치는 풀은 유동성이 더 깊을 수 있다.

그래서 라우터는 여러 경로를 비교할 수 있다.

```text
A -> B
A -> WETH -> B
A -> USDC -> B
A -> WETH -> USDC -> B
```

다만 hop이 많다고 무조건 좋은 것은 아니다. 더 많은 풀을 지나면 더 좋은 가격을 찾을 수도 있지만, gas 비용이 늘고, 경로가 길어지면서 실패 가능성이나 가격 변동 영향도 커질 수 있다.

## Router는 경로를 찾아주는 역할이다

사용자는 보통 "ETH를 USDC로 바꾸고 싶다" 정도만 입력한다. 그 다음 어떤 풀을 지나야 하는지, 직접 경로가 좋은지, 중간 토큰을 거치는 경로가 좋은지는 라우팅 로직이 계산한다.

이때 나오는 단어가 router다.

라우터는 사용자의 스왑 요청을 실제 풀 호출로 연결해주는 역할을 한다. 개발 문서에서는 path, route, hop 같은 단어가 같이 나온다.

처음에는 이렇게 나눠서 보면 편하다.

| 용어 | 의미 |
| --- | --- |
| path | 토큰이 지나가는 순서 |
| route | 실제 스왑 경로 |
| hop | 경로 안에서 풀 하나를 지나는 단계 |
| router | 적절한 경로로 스왑을 실행하도록 돕는 컨트랙트나 로직 |

예를 들어 아래 path는 두 번의 hop을 가진다.

```text
USDC -> WETH -> UNI
```

이런 경로를 이해하면 `hop`은 더 이상 낯선 단어가 아니다. 그냥 "중간에 몇 개의 풀을 지나갔는가"를 세는 말에 가깝다.

## Approve와 Swap은 다른 트랜잭션일 수 있다

Uniswap을 처음 사용할 때 또 헷갈리는 부분은 `approve`다.

ERC-20 토큰을 스왑하려면, 사용자가 먼저 Uniswap 쪽 컨트랙트가 내 토큰을 사용할 수 있도록 허용해야 하는 경우가 많다. 이 허용을 approve라고 부른다.

흐름은 이렇게 나뉠 수 있다.

```text
approve
-> 특정 컨트랙트가 내 토큰을 쓸 수 있도록 허용

swap
-> 실제로 토큰을 교환
```

approve는 로그인 버튼이 아니다. 내 토큰을 일정 범위 안에서 사용할 권한을 주는 온체인 상태 변경이다.

그래서 어떤 토큰을 어떤 컨트랙트에 얼마만큼 허용하는지 확인해야 한다. 특히 처음 보는 사이트에서 무제한 approve를 요구하면 더 조심해야 한다.

Uniswap 자체를 공부할 때도 이 구분은 중요하다.

```text
토큰 사용 권한을 주는 일과
실제로 스왑하는 일은 다를 수 있다.
```

## Uniswap을 볼 때 조심할 것

Uniswap은 유용하지만, 초심자가 아무 생각 없이 쓰기에는 조심할 부분도 많다.

- 토큰 티커만 보고 진짜 토큰이라고 믿으면 안 된다.
- contract address와 chain을 확인해야 한다.
- 슬리피지를 너무 높게 잡으면 불리한 체결이 될 수 있다.
- 유동성이 낮은 풀에서는 price impact가 커질 수 있다.
- approve 권한은 필요할 때만 주고, 큰 권한은 주기적으로 점검하는 편이 좋다.
- 지갑 팝업에서 어떤 컨트랙트와 상호작용하는지 확인해야 한다.

특히 토큰은 [[ticker-token-symbol|티커]]만으로 식별하면 위험하다. 같은 심볼을 가진 다른 토큰이 있을 수 있고, 가짜 토큰이 유명한 이름을 흉내 낼 수도 있다.

DEX에서는 사용자가 직접 지갑으로 서명한다. 이 말은 중간 관리자 없이 빠르게 행동할 수 있다는 뜻이기도 하지만, 잘못된 토큰이나 잘못된 승인도 사용자가 직접 실행할 수 있다는 뜻이다.

## 처음 실습할 때 볼 것

처음부터 큰 금액으로 스왑할 필요는 없다. 오히려 작은 금액이나 테스트 환경에서 화면을 천천히 읽는 편이 좋다.

Uniswap을 볼 때는 아래 순서로 확인해보면 도움이 된다.

1. 어떤 chain에서 스왑하는지 본다.
2. 입력 토큰과 출력 토큰의 contract address를 확인한다.
3. 예상 수령량과 최소 수령량을 나눠서 본다.
4. price impact가 큰지 확인한다.
5. route나 path가 표시된다면 중간 토큰을 거치는지 본다.
6. approve 요청인지 swap 요청인지 구분한다.
7. 트랜잭션이 성공한 뒤 블록 탐색기에서 `from`, `to`, `input data`, `logs`를 확인한다.

이 실습에서 중요한 질문은 이것이다.

```text
내 지갑은 어떤 컨트랙트에 무엇을 허용했는가?
스왑은 어떤 풀을 지나갔는가?
결과적으로 어떤 토큰 잔액이 바뀌었는가?
```

이 질문에 답할 수 있으면 Uniswap은 단순히 "토큰 바꾸는 사이트"가 아니라, 지갑과 스마트 컨트랙트와 유동성 풀이 이어진 온체인 시스템으로 보이기 시작한다.

## 정리

Uniswap은 토큰을 교환하는 DEX 프로토콜이다. 하지만 그 안을 보면 단순한 환전 서비스라기보다 스마트 컨트랙트, 유동성 풀, AMM, 라우터가 이어진 구조에 가깝다.

처음에는 이렇게만 잡아도 충분하다.

```text
Uniswap은 주문서 대신 유동성 풀을 쓴다.
사용자는 지갑으로 트랜잭션에 서명한다.
AMM은 풀의 상태를 기준으로 교환 비율을 계산한다.
Router는 어떤 경로로 스왑할지 찾는다.
Hop은 그 경로에서 풀 하나를 지나는 단계다.
```

결국 `hop`이라는 작은 단어도 Uniswap의 구조를 보여준다.

토큰 스왑은 단순히 A를 B로 바꾸는 버튼이 아니다. 여러 유동성 풀 사이에서 어떤 길로 가야 원하는 토큰을 받을 수 있는지 찾는 과정이다.

그래서 Uniswap을 처음 공부할 때는 가격 차트보다 경로를 먼저 보는 편이 좋다.

```text
내 토큰은 어떤 풀을 지나 어디로 갔는가?
```

이 질문이 잡히면 `swap`, `route`, `hop` 같은 단어가 하나의 흐름 안에 들어오기 시작한다.
