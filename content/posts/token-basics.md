![토큰(Token) 이해하기](/images/token-basics-cover.png)

블록체인을 공부하다 보면 `토큰`, `코인`, `ERC-20`, `NFT`, `approve`, `swap` 같은 단어가 한꺼번에 나온다.

처음에는 토큰을 "코인 같은 것" 정도로 생각하기 쉽다. 그런데 그렇게만 보면 금방 헷갈린다. 어떤 곳에서는 토큰이 돈처럼 쓰이고, 어떤 곳에서는 게임 포인트처럼 쓰이고, 또 어떤 곳에서는 투표권이나 멤버십처럼 설명되기 때문이다.

나도 처음에는 이 부분이 잘 붙지 않았다. 토큰이 지갑 안에 들어 있는 파일 같은 것인지, 블록체인 위에 따로 떠다니는 물건인지, 아니면 그냥 화면에 표시되는 숫자인지 애매했다.

이 글은 토큰을 깊은 금융 상품이나 프로토콜 관점으로 설명하는 글은 아니다. [[web3-service-basics|Web3 기반 서비스]]를 이해할 때 자주 막히는 "토큰이 정확히 어디에 있고, 무엇이 바뀌는가?"라는 질문을 초심자 관점에서 정리하는 메모에 가깝다.

핵심 질문은 세 개다.

```text
토큰은 어디에 기록될까?
지갑은 토큰을 실제로 담고 있을까?
토큰을 보낸다는 것은 무엇이 바뀐다는 뜻일까?
```

이 질문이 잡히면 ERC-20, swap, approve 같은 다음 개념도 훨씬 덜 낯설어진다.

## 먼저 한 문장으로 잡기

토큰을 한 문장으로 줄이면 이렇게 볼 수 있다.

```text
토큰은 블록체인 상태에 기록된 주소별 수량이나 권리다.
```

조금 더 풀면 이렇다.

```text
어떤 주소가
어떤 프로젝트의 토큰을
얼마나 가지고 있는지
블록체인이나 컨트랙트 장부에 기록한 것
```

예를 들어 어떤 `ABC` 토큰이 있다면 토큰 컨트랙트 안에는 이런 기록이 있을 수 있다.

```text
0xAAA -> 100 ABC
0xBBB -> 20 ABC
0xCCC -> 0 ABC
```

이때 `0xAAA` 주소가 `ABC` 토큰 100개를 가진다고 말한다.

중요한 점은 토큰이 지갑 앱 안에 파일처럼 들어 있는 것이 아니라는 점이다. 실제 기록은 블록체인 상태에 있고, 지갑은 그 상태를 조회하거나 바꾸기 위한 서명을 만들어주는 도구에 가깝다.

## 토큰은 어디에 있을까

토큰을 처음 볼 때 가장 먼저 버려야 하는 그림은 이것이다.

```text
지갑 안에 토큰이 들어 있다
```

현실의 지갑에는 현금이나 카드가 들어 있다. 그래서 블록체인 지갑에도 코인이나 토큰이 들어 있을 것처럼 느껴진다. 하지만 이더리움 계열 지갑을 조금 더 정확히 보면 다르다.

![토큰은 어디에 있을까?](/images/token-contract-ledger-handdrawn.png)

지갑은 보통 이런 일을 한다.

```text
개인키를 보관한다.
주소를 보여준다.
블록체인 상태를 조회한다.
사용자가 승인하면 트랜잭션에 서명한다.
```

반면 토큰 잔액 기록은 보통 토큰 컨트랙트가 들고 있다.

개발자 관점에서는 아주 단순하게 이런 자료구조를 떠올리면 된다.

```text
Map<Address, Balance>
```

예를 들어 ERC-20 토큰 컨트랙트 안에는 주소별 잔액이 기록된다.

```text
balances[0xAAA] = 100
balances[0xBBB] = 20
balances[0xCCC] = 0
```

그래서 내 지갑 앱에서 `ABC 100개`가 보인다는 말은, 지갑 앱 안에 ABC 파일 100개가 들어 있다는 뜻이 아니다.

더 정확히는 이런 뜻이다.

```text
ABC 토큰 컨트랙트의 장부를 조회해보니
내 주소에 100이라는 잔액이 기록되어 있다.
```

이 관점 하나만 잡아도 많은 표현이 덜 헷갈린다.

```text
토큰을 받았다
= 토큰 컨트랙트 장부에서 내 주소의 잔액이 늘었다.

토큰을 보냈다
= 토큰 컨트랙트 장부에서 내 주소의 잔액이 줄고, 상대 주소의 잔액이 늘었다.

토큰 잔액을 조회했다
= 토큰 컨트랙트 장부에서 내 주소에 연결된 숫자를 읽었다.
```

## 그럼 코인과 토큰은 뭐가 다를까

토큰을 이해할 때 자주 같이 나오는 말이 코인이다.

둘을 엄밀하게 나누는 방식은 생태계마다 조금씩 다르게 쓰일 수 있다. 그래도 초심자 관점에서는 이렇게 나누면 꽤 유용하다.

```text
코인:
그 블록체인 네트워크의 기본 자산

토큰:
특정 컨트랙트나 규칙이 만들어낸 자산 또는 권리 단위
```

Ethereum 네트워크에서는 `ETH`가 네이티브 코인이다. ETH는 이더리움 계정 상태의 기본 잔액으로 다뤄진다.

반면 `ABC`라는 ERC-20 토큰이 있다면, 이 토큰의 잔액은 보통 `ABC` 토큰 컨트랙트 안의 기록으로 다뤄진다.

![코인과 토큰은 기록되는 위치가 다르다](/images/native-coin-vs-token-handdrawn.png)

아주 단순하게 비교하면 이렇다.

| 구분 | 네이티브 코인 | 토큰 |
| --- | --- | --- |
| 예시 | ETH, MATIC, SOL | USDC, UNI, 게임 토큰 |
| 기록 위치 | 체인의 기본 계정 상태 | 토큰 컨트랙트의 상태 |
| 전송 방식 | 네트워크 기본 전송 | 컨트랙트 함수 호출 |
| 수수료 | 보통 이 코인으로 지불 | 전송하려면 별도 가스 코인이 필요할 수 있음 |

예를 들어 Ethereum에서 USDC를 보낸다고 해보자.

이때 수수료는 보통 ETH로 낸다. 보내는 자산은 USDC지만, 트랜잭션을 실행하기 위한 가스비는 네트워크의 기본 코인인 ETH로 지불한다.

그래서 지갑에서 이런 상황을 볼 수 있다.

```text
USDC는 충분한데 ETH가 없어서 전송이 안 된다.
```

처음에는 이상해 보인다. 하지만 기록 위치와 수수료 자산을 나눠서 보면 이해된다.

```text
USDC:
토큰 컨트랙트 안의 잔액 기록

ETH:
Ethereum 네트워크에서 트랜잭션을 실행하기 위한 기본 코인
```

## 토큰 전송은 장부 숫자를 바꾸는 일

토큰을 보낸다고 하면 토큰 칩이나 디지털 물건이 내 지갑에서 상대 지갑으로 날아가는 그림을 떠올리기 쉽다. 설명용 비유로는 괜찮지만, 실제로 바뀌는 것은 장부의 숫자다.

![토큰 전송은 장부 숫자를 바꾸는 일](/images/token-transfer-ledger-update-handdrawn.png)

예를 들어 A가 B에게 `ABC` 토큰 10개를 보낸다고 해보자.

전송 전 상태는 이렇다.

```text
A 주소: 100 ABC
B 주소: 20 ABC
```

A의 지갑이 토큰 컨트랙트에 `transfer(B, 10)` 같은 요청을 보내고, 네트워크가 이 트랜잭션을 유효하다고 받아들이면 상태가 바뀐다.

```text
A 주소: 90 ABC
B 주소: 30 ABC
```

이 흐름을 조금 더 풀면 이렇다.

```text
1. 사용자가 지갑에서 토큰 전송을 요청한다.
2. 지갑이 전송 내용을 보여준다.
3. 사용자가 승인하면 지갑이 개인키로 트랜잭션에 서명한다.
4. 서명된 트랜잭션이 네트워크로 전송된다.
5. 노드들이 서명과 규칙을 검증한다.
6. 토큰 컨트랙트의 transfer 규칙이 실행된다.
7. 컨트랙트 장부의 주소별 잔액이 바뀐다.
```

여기서 지갑이 하는 일은 중요하지만, 지갑이 혼자 장부를 바꾸는 것은 아니다.

지갑은 사용자의 권한을 증명하는 서명을 만든다. 컨트랙트는 정해진 규칙대로 상태를 바꾼다. 네트워크는 그 상태 변경이 유효한지 검증하고 공유한다.

그래서 토큰 전송을 아주 짧게 줄이면 이렇게 볼 수 있다.

```text
내 주소의 권한으로
토큰 컨트랙트의 장부 숫자를 바꾸는 트랜잭션
```

이 관점은 [[simple-payment-blockchain-signature|블록체인 서명]]을 이해할 때도 중요하다. 지갑에서 승인 버튼을 누르는 것은 단순한 화면 확인이 아니라, 특정 상태 변경을 내 주소의 권한으로 실행하겠다는 서명이다.

## 토큰은 하나의 용도만 뜻하지 않는다

토큰이 어려운 이유는 토큰이라는 말이 너무 넓게 쓰이기 때문이다.

어떤 토큰은 달러 같은 실제 자산 가치에 맞춰 쓰이려고 한다. 어떤 토큰은 게임 안의 재화처럼 쓰인다. 어떤 토큰은 프로젝트 의사결정에 참여하는 투표권처럼 쓰인다. 어떤 토큰은 특정 멤버십이나 이벤트 티켓을 나타낼 수도 있다.

![토큰은 하나의 용도만 뜻하지 않는다](/images/token-use-cases-handdrawn.png)

예를 들면 이런 식이다.

| 종류 | 토큰이 나타내는 것 |
| --- | --- |
| 스테이블코인 | 특정 화폐 가치에 연동되도록 설계된 자산 |
| 게임 토큰 | 게임 안의 보상, 아이템 구매, 포인트 |
| 거버넌스 토큰 | 프로젝트 의사결정에 참여할 수 있는 권리 |
| 멤버십 토큰 | 특정 혜택, 접근 권한, 이벤트 참여 자격 |
| NFT | 고유한 소유권이나 식별 가능한 디지털 항목 |

그래서 토큰을 볼 때는 먼저 이렇게 물어보는 편이 좋다.

```text
이 토큰은 무엇을 나타내는가?
누가 발행했는가?
어떤 컨트랙트 주소를 가지는가?
어떤 네트워크 위에 있는가?
어디에서 쓸 수 있는가?
```

토큰이라는 단어만으로는 부족하다. 토큰은 그 프로젝트가 정한 규칙과 맥락 안에서 의미를 가진다.

## 대체 가능한 토큰과 NFT

ERC-20을 이해하려면 "대체 가능하다"는 감각도 필요하다.

대체 가능하다는 것은 같은 단위끼리 서로 구분할 필요가 없다는 뜻이다.

예를 들어 1 USDC와 다른 1 USDC는 보통 서로 구분하지 않는다. 내 지갑의 1 USDC와 다른 사람 지갑의 1 USDC가 같은 컨트랙트의 같은 단위라면, 둘은 같은 1개로 취급된다.

이런 토큰을 대체 가능한 토큰이라고 부른다.

```text
1 ABC = 다른 1 ABC
10 ABC = 1 ABC 10개
```

반면 NFT는 보통 각각이 다르다.

```text
tokenId 7 NFT != tokenId 8 NFT
```

둘 다 "토큰"이라는 말을 쓰지만, 성격이 다르다.

| 구분 | 대체 가능한 토큰 | NFT |
| --- | --- | --- |
| 대표 표준 | ERC-20 | ERC-721 |
| 단위 | 같은 1개끼리 구분하지 않음 | 각각의 tokenId가 고유함 |
| 예시 | 포인트, 스테이블코인, 거버넌스 토큰 | 그림, 게임 아이템, 티켓, 멤버십 카드 |
| 기록 | 주소별 수량 | tokenId별 소유자 |

초심자 단계에서는 이렇게만 잡아도 충분하다.

```text
ERC-20 계열 토큰:
주소별로 몇 개를 가지고 있는지 본다.

NFT:
특정 tokenId를 누가 가지고 있는지 본다.
```

이 차이는 [[minting-basics|민팅]]을 볼 때도 이어진다. ERC-20 민팅은 보통 어떤 주소의 수량과 전체 공급량이 늘어나는 흐름이고, NFT 민팅은 특정 tokenId의 소유자가 새로 기록되는 흐름에 가깝다.

## 스마트 컨트랙트가 왜 나올까

토큰을 설명하다 보면 결국 스마트 컨트랙트가 나온다. 이름은 어렵지만 여기서는 이렇게 보면 된다.

```text
토큰 컨트랙트
= 토큰의 잔액과 전송 규칙을 들고 있는 블록체인 프로그램
```

ERC-20 토큰이라면 보통 이런 기능을 제공한다.

```text
balanceOf(address)
-> 이 주소가 토큰을 얼마나 가지고 있는지 조회한다.

transfer(to, amount)
-> 내 주소의 토큰을 다른 주소로 보낸다.

approve(spender, amount)
-> 다른 주소나 컨트랙트가 내 토큰을 정해진 범위 안에서 쓸 수 있게 허용한다.

transferFrom(from, to, amount)
-> 허용받은 범위 안에서 토큰을 대신 옮긴다.
```

여기서 `approve`는 특히 조심해야 한다.

처음 보면 "승인"이라는 단어 때문에 로그인이나 단순 확인처럼 느껴질 수 있다. 하지만 ERC-20에서 approve는 내 토큰을 누군가가 대신 사용할 수 있도록 권한을 주는 상태 변경이다.

예를 들어 DEX에서 토큰을 [[swap-token-exchange|swap]]할 때는 이런 흐름이 자주 나온다.

```text
1. DEX 컨트랙트가 내 ABC 토큰을 사용할 수 있도록 approve한다.
2. DEX 컨트랙트가 허용된 범위 안에서 ABC를 가져가고, 다른 토큰을 돌려준다.
```

그래서 지갑에서 approve 요청이 뜨면 아래를 봐야 한다.

```text
어떤 토큰에 대한 권한인가?
어떤 컨트랙트에게 권한을 주는가?
얼마나 허용하는가?
무제한 허용인가?
신뢰할 수 있는 서비스인가?
```

토큰을 장부 기록으로 이해하면 approve도 조금 덜 모호해진다.

```text
approve는 토큰을 바로 보내는 일이 아니라,
컨트랙트 장부에 사용 허용량을 기록하는 일이다.
```

## 티커만 보고 토큰을 믿으면 안 된다

지갑이나 거래소 화면에서는 보통 `USDC`, `ABC`, `WEMIX` 같은 짧은 이름을 본다. 이런 짧은 표시 이름을 토큰 심볼이나 [[ticker-token-symbol|티커]]라고 부른다.

하지만 티커는 토큰의 진짜 신분증이 아니다.

누구나 `ABC`라는 심볼을 가진 토큰 컨트랙트를 만들 수 있다. 그래서 같은 네트워크 안에도 같은 티커를 가진 다른 토큰이 있을 수 있고, 다른 네트워크까지 포함하면 더 많아진다.

초심자에게 중요한 구분은 이것이다.

```text
티커:
사람이 보기 쉬운 짧은 표시 이름

컨트랙트 주소:
특정 네트워크 위에서 토큰 컨트랙트를 가리키는 주소
```

그래서 토큰을 확인할 때는 티커만 보지 말고 네트워크와 컨트랙트 주소를 같이 봐야 한다.

```text
Ethereum 위의 ABC 토큰
Polygon 위의 ABC 토큰
내가 아는 프로젝트의 공식 ABC 토큰
누군가 만든 가짜 ABC 토큰
```

화면에서는 전부 `ABC`처럼 보일 수 있다. 하지만 실제로는 서로 다른 토큰일 수 있다.

## 지갑 개발자 관점에서 다시 보기

지갑 앱을 개발하는 입장에서는 토큰을 "잔액 숫자"로만 보면 부족하다. 사용자가 보는 화면 뒤에는 여러 조회와 검증이 붙는다.

예를 들어 토큰 목록을 보여주려면 대략 이런 정보가 필요하다.

```text
네트워크
토큰 컨트랙트 주소
토큰 이름
토큰 심볼
decimals
사용자 주소의 balance
가격 정보가 필요하다면 별도 가격 API
```

여기서 `decimals`도 자주 헷갈린다.

블록체인 컨트랙트 안에서는 보통 정수로 수량을 저장한다. 사람이 보기 좋게 소수점 위치를 정하는 값이 `decimals`다.

예를 들어 `decimals = 18`인 토큰에서 컨트랙트 내부 값이 아래처럼 저장될 수 있다.

```text
1000000000000000000
```

화면에서는 이 값을 `1.0 TOKEN`처럼 보여준다.

그래서 지갑 앱은 단순히 숫자를 가져와서 그대로 보여주면 안 된다. 토큰의 decimals를 적용해 사람이 읽을 수 있는 형태로 바꿔야 한다.

그리고 토큰 전송 화면에서는 더 많은 것을 보여줘야 한다.

```text
어떤 네트워크인가?
어떤 토큰인가?
받는 주소는 무엇인가?
보내는 수량은 얼마인가?
수수료로 필요한 네이티브 코인은 충분한가?
이 토큰이 공식 토큰이 맞는가?
```

사용자에게는 "토큰 보내기" 버튼 하나지만, 앱은 뒤에서 네트워크, 컨트랙트, 주소, 수량 단위, 가스비를 모두 맞춰야 한다.

## 한 번에 정리하기

토큰을 처음 이해할 때는 가격이나 차트보다 기록 위치를 먼저 보는 편이 좋다.

```text
토큰은 지갑 안에 들어 있는 물건이 아니다.
토큰은 블록체인이나 컨트랙트 상태에 기록된 주소별 수량 또는 권리다.
지갑은 그 기록을 조회하고, 사용자의 서명으로 상태 변경을 요청한다.
토큰 전송은 컨트랙트 장부의 잔액 숫자를 바꾸는 트랜잭션이다.
토큰의 의미는 프로젝트가 정한 규칙과 용도에 따라 달라진다.
```

이 정도가 잡히면 ERC-20도 더 쉽게 보인다.

ERC-20은 "토큰은 이런 식으로 잔액을 조회하고, 전송하고, 다른 컨트랙트에게 사용 권한을 줄 수 있어야 한다"는 이더리움 토큰 표준이다.

즉 다음 글에서 볼 ERC-20은 완전히 새로운 개념이라기보다, 지금 본 토큰 장부의 공통 인터페이스에 가깝다.

```text
토큰
= 블록체인 상태에 기록된 주소별 수량이나 권리

ERC-20
= 이더리움에서 대체 가능한 토큰을 다루기 위한 공통 규칙
```

처음 목표는 모든 토큰 표준을 외우는 것이 아니다. 지갑 화면에서 토큰 잔액을 볼 때, "이 숫자는 어디에서 온 것인가?"를 설명할 수 있으면 된다.

그 질문에 답할 수 있으면 토큰은 더 이상 막연한 코인 비슷한 것이 아니라, 컨트랙트와 지갑과 트랜잭션 사이에 놓인 구체적인 상태 기록으로 보이기 시작한다.
